🌐 Загрози мультимедійним сервісам

VoIP/UC-системи поєднують риси веб-застосунків, мережевих протоколів та телефонії — і успадковують вразливості всіх трьох рівнів.

🔴 Доступність

  • DoS / DDoS
  • SIP flood
  • Resource exhaustion
  • Fuzzing / malformed packets

🟡 Фінансові

  • Toll Fraud / IRSF
  • Компрометація транків
  • SPIT (небажані дзвінки)
  • Викрадення послуг зв'язку

🔵 Конфіденційність

  • Eavesdropping (прослуховування)
  • Man-in-the-Middle (MitM)
  • Розкриття метаданих
  • Злам реєстрації SIP
За даними ENISA (звіт 2023 р.): щорічні збитки від телефонного фроду у світі — понад 40 млрд USD; DDoS на VoIP-оператори зростають на ~30% щороку (дані можуть бути оновлені у нових звітах).

💥 DoS / DDoS — Огляд

Мета: зробити VoIP-сервіс недоступним для легітимних користувачів шляхом виснаження обчислювальних, мережевих або сигнальних ресурсів.

Типи атак

  • Volumetric — переповнення каналу (UDP/ICMP flood)
  • Protocol — експлуатація вразливостей протоколу (SYN flood, SIP flood)
  • Application-layer — виснаження ресурсів застосунку (INVITE flood, REGISTER flood)
  • Resource exhaustion — переповнення чергів, пам'яті, CPU медіа-сервера

Особливості VoIP-середовища

  • SIP — текстовий, легко генерувати запити
  • UDP — без рукостискання → ефективний спуфінг
  • Stateful сервери (proxy, B2BUA) — вразливі до state exhaustion
  • Медіа-сервери (RTP mixer, transcoder) — обмежені CPU/пам'ять
  • Жорсткі вимоги до затримки (<150 мс) → навіть 30% падіння = відмова

📨 SIP Flood — Типи та механізми

INVITE Flood

  • Масова відправка INVITE-запитів на proxy/PBX
  • Сервер виділяє стан транзакції для кожного запиту
  • Відповіді 100 Trying → виснаження пам'яті
  • Типово 10 000+ INVITE/сек вистачає для збою Asterisk на стандартному сервері (реальне значення залежить від версії, апаратного забезпечення та конфігурації)

REGISTER Flood

  • Лавинна реєстрація фіктивних UA
  • Виснаження бази даних реєстрацій
  • Блокування легітимних REGISTER-запитів

OPTIONS / NOTIFY Flood

  • OPTIONS — keep-alive метод, легко генерувати
  • NOTIFY flood — атака на підписки (SUBSCRIBE)
  • Не потребують попередньої сесії

Malformed SIP

  • Надмірно великі заголовки (Via, Contact)
  • Некоректний SDP body
  • Мета: crash / buffer overflow сервера
  • Fuzzing: SIPp, Peach, custom scripts

🔬 SIP INVITE Flood — технічні деталі

;; Приклад мінімального INVITE для flood-атаки (генерується тисячами)
INVITE sip:100@192.168.1.10 SIP/2.0
Via: SIP/2.0/UDP 10.0.0.1:5060;branch=z9hG4bK-
From: <sip:attacker@evil.com>;tag=<RANDOM>
To: <sip:100@192.168.1.10>
Call-ID: <RANDOM>@10.0.0.1
CSeq: 1 INVITE
Content-Type: application/sdp
Content-Length: 135

v=0
o=attacker 0 0 IN IP4 10.0.0.1
s=-
c=IN IP4 10.0.0.1
t=0 0
m=audio 10000 RTP/AVP 0
a=rtpmap:0 PCMU/8000
Що відбувається на сервері: кожен INVITE → нова транзакція у стані Proceeding → виділення пам'яті (~5–20 КБ) → таймер B (32 с) → виснаження RAM/CPU
Інструменти атаки: SIPp (-sn uac), inviteflood, sipsak, sipflood (Python), Metasploit sip_invite_spoof

🎙️ Атаки на ресурси медіа-серверів

RTP Flood

  • Лавинна відправка UDP-пакетів на медіа-порти (динамічні, 10000–65535)
  • Медіа-сервер обробляє кожен пакет → CPU overload
  • Mix/transcoding pipeline зупиняється
  • Наслідок: хрипи, розриви, відключення активних дзвінків
RTP не має автентифікації → будь-який хост може надсилати потік на відкритий порт

Інші вектори

Conferencing exhaustion

  • Масове приєднання до конференц-бриджів
  • Виснаження ліцензій та потоків мікшування

Transcoding CPU Attack

  • Примусовий транскодинг (G.711 ↔ Opus, SILK)
  • CPU-інтенсивна операція × 1000 потоків = відмова
  • Актуально для WebRTC-шлюзів

📡 DDoS-ампліфікація у VoIP-середовищі

Зловмисник                   Відкритий SIP Proxy          Жертва
    │                                │                        │
    │  OPTIONS sip:victim@proxy      │                        │
    │  (src IP = IP_жертви, спуфінг) │                        │
    │───────────────────────────────►│                        │
    │                                │  200 OK (більший пакет)│
    │                                │───────────────────────►│
    │   × 10 000 запитів/сек         │  × 10 000 відповідей   │
    │                                │  (ампліфікація ~3–8x)  │

SIP-ампліфікація

  • Відкриті SIP proxy відповідають більшим пакетом ніж запит
  • Коефіцієнт ампліфікації: 3–8x (відношення розміру відповіді до розміру запиту)
  • Мільйони відкритих SIP proxy у мережі Інтернет (Shodan)

DNS/NTP ампліфікація проти VoIP

  • Класичні ампліфікатори (DNS ×50, NTP ×556) направляються на VoIP-інфраструктуру
  • SIP-сервер не може відрізнити від легітимного UDP-трафіку
  • Захист: BCP38, rate limiting, anycast

🛡️ Захист від DoS/DDoS атак

На рівні SIP proxy / PBX

  • Rate limiting — обмеження INVITE/REGISTER per IP/user (fail2ban, Kamailio pike)
  • Stateless ACK — мінімальний стан для незавершених транзакцій
  • SIP TLS / digest auth — відфільтровує анонімних атакуючих
  • IP whitelist — дозволяти лише відомі SBC/trunk
  • Regex-фільтрація заголовків (malformed SIP)
  • SIP SBC (Session Border Controller) — перший рубіж оборони

На мережевому рівні

  • Anti-spoofing (BCP38/BCP84) — блокування підроблених джерел
  • Scrubbing center / Cloud DDoS mitigation
  • QoS + DSCP marking — пріоритет реальних медіа-потоків
  • Firewall + stateful inspection для SIP/RTP
  • RTCP feedback — моніторинг аномалій потоку
  • Anomaly detection (ML) — відхилення від baseline
Ключовий інструмент: Session Border Controller (SBC) — єдина точка входу, яка реалізує rate limiting, TLS termination, topology hiding та SIP normalization.

📞 SPIT — Spam over IP Telephony

SPIT — масове автоматичне надсилання небажаних голосових дзвінків через VoIP-мережі. Аналог email-спаму, але в телефонії.

Чому SPIT — реальна загроза?

  • Собівартість VoIP-дзвінка близька до нуля
  • Один сервер може генерувати тисячі дзвінків/хв
  • SIP не вимагає автентифікації відправника
  • Caller ID легко підробляється (spoofing)
  • Обходить PSTN-засоби захисту
  • Складно фільтрувати: голос = реальний час

Сценарії використання

  • Комерційний SPIT — рекламні дзвінки, robocalls
  • Vishing — телефонний фішинг (банки, держ. установи)
  • Wangiri — дзвінки на 1 гудок (перетворення дзвінка у toll fraud)
  • Political robocall — масова агітація
  • DoS через SPIT — перевантаження call-центрів

⚙️ Механізми масових небажаних дзвінків

 SPIT-Ботнет / VoIP-сервер зловмисника
         │
         │  INVITE (підроблений Caller ID = "+380(XX) банк")
         ▼
  SIP Proxy жертви / оператора
         │
         │  INVITE → UA Жертви
         ▼
  ┌────────────┐      Жертва піднімає трубку
  │  UA Жертви │ ◄── відтворюється записане повідомлення
  └────────────┘      або живий шахрай (vishing)

Технічна реалізація SPIT

  • Asterisk / FreeSWITCH + скрипт dialplan
  • SIP trunk з дешевим VoIP-провайдером
  • Підробка From: та P-Asserted-Identity
  • Автодозвон зі списку номерів
  • TTS або заздалегідь записане аудіо

Масштаб атаки

  • 1 Asterisk-сервер → ~200–500 паралельних дзвінків
  • Ботнет з 100 серверів → 20 000–50 000 одночасних дзвінків
  • За добу — мільйони абонентів
  • Wangiri: дзвінок 1–2 сек → жертва перетелефоновує на платний номер

🔍 Виявлення SPIT

Поведінковий аналіз

  • Висока частота коротких дзвінків
  • Низький показник відповідей (call completion rate)
  • Незвичні години активності
  • Однаковий CDR-шаблон
  • Аномальна кількість унікальних одержувачів

Аналіз сигналізації

  • Повторюваний User-Agent
  • Відсутність P-Asserted-Identity
  • Незареєстрований From:
  • Незмінний Contact: IP
  • Низька варіативність Call-ID

Репутаційні системи

  • STIR/SHAKEN — криптографічна атестація Caller ID
  • Blacklist/whitelist баз номерів
  • Community-based reputation (Hiya, YouMail)
  • ML-класифікація за CDR патернами
STIR/SHAKEN (RFC 8224, RFC 8226): оператор підписує Identity-заголовок SIP за допомогою сертифіката — одержувач може перевірити справжність номера відправника.

🛡️ Методи протидії SPIT

Мережевий / протокольний рівень

  • Обов'язкова SIP-автентифікація (digest, TLS клієнтський сертифікат)
  • Rate limiting дзвінків per source IP/user
  • CAPTCHA / Turing test для нових UA (IVR challenge)
  • STIR/SHAKEN — верифікація Caller ID через PKI
  • Грейлістинг — затримка першого дзвінку з нового джерела

Рівень застосунку / оператора

  • Consent-based calling — дзвінки лише до підтверджених абонентів
  • Blacklist/Whitelist на рівні SBC та PBX
  • Call analytics — виявлення аномальних шаблонів CDR
  • Spam score — ML-модель оцінки ризику до підняття трубки
  • Legal/regulatory — реєстрація robocall-відправників
Повного захисту від SPIT не існує — ефективний захист потребує комбінації автентифікації, репутаційних систем та поведінкового аналізу.

💰 Toll Fraud — Вступ

Toll Fraud — незаконне використання телекомунікаційних послуг за рахунок жертви: підприємства, оператора або абонента.

Масштаб проблеми

  • CFCA (Communications Fraud Control Association, звіт 2023 р.): ~$40 млрд/рік
  • Найпоширеніший вид телеком-злочину
  • Часто виявляється лише після виставлення рахунку
  • За кілька годин можуть бути здійснені дзвінки на $10 000+

Основні схеми

  • IRSF — International Revenue Share Fraud
  • PBX Hacking — злам корпоративної АТС
  • Trunk Hijacking — компрометація SIP-транку
  • Subscription Fraud — реєстрація послуг на підставні особи
  • Wangiri — зворотній дзвінок на Premium Rate
  • SIM Boxing — арбітраж VoIP ↔ PSTN

     │  Злам PBX / SIP-облікових   │                                │
     │  даних жертви               │                                │

     │  INVITE → +88216XXXXXXX    │                                │

     │   Diego Garcia і т.д.)      │                                │
     │────────────────────────────►│──────────────────────────────►│

Механізм: зловмисник отримує Premium Rate Number у екзотичній юрисдикції, зламує PBX жертви, генерує трафік → частка тарифу повертається зловмиснику
Типові напрямки IRSF: Vanuatu (+678), Sao Tome (+239), Diego Garcia (+246), Latvia Premium (+371 9XX), Satellite числа

🔓 Компрометація транків та PBX

Вектори атаки на SIP-транк

  • Brute-force SIP credentials — перебір логін/пароль реєстрації
  • Default credentials — стандартні паролі IP PBX (admin/admin, 1234)
  • Слабкий digest auth — MD5 rainbow tables
  • SIP trunk без автентифікації (тільки IP ACL)
  • Web UI RCE — атака на адмін-панель Asterisk, FreePBX, 3CX
  • Phishing адміністратора — зміна паролю транку

Після компрометації

Що робить зловмисник

  • Генерує IRSF-трафік через транк жертви
  • Встановлює backdoor у dialplan
  • Перенаправляє вхідні дзвінки (eavesdropping)
  • Використовує PBX як плацдарм для SPIT
Інструменти: SIPVicious (svcrack, svmap), Hydra, sipcrack, Metasploit SIP modules

🛡️ Захист від Toll Fraud

Технічні заходи

  • Надійні паролі SIP — мінімум 16 символів, rotate щоквартально
  • IP ACL — дозволяти транк лише з IP провайдера
  • TLS/SIPS — захист credentials від перехоплення
  • Обмеження напрямків — блокувати Premium Rate, міжнародні на рівні dialplan
  • Call limit per extension — максимум паралельних дзвінків
  • Fail2ban для SIP — блокування після N невдалих спроб авт.
  • Моніторинг CDR у реальному часі + alert

Організаційні заходи

  • Fraud Management System (FMS) оператора — автоматичне відключення
  • Ліміти витрат на акаунт/транк
  • Регулярний аудит dialplan та users
  • Оновлення ПЗ PBX — патчі вразливостей
  • Розмежування мереж — PBX в окремому VLAN
  • IRSF blacklist — міжнародні бази підозрілих номерів (i3forum)

👂 Eavesdropping — Прослуховування

Eavesdropping — несанкціонований перехват сигналізації та/або медіа-потоків VoIP-комунікацій.

Що може бути перехоплено

  • SIP-сигналізація — INVITE, BYE, облікові дані (digest)
  • SDP — порти та кодеки медіа
  • RTP-потоки — голос/відео у відкритому вигляді
  • DTMF — PIN-коди, паролі IVR
  • Метадані — хто, коли, куди, тривалість

Умови для перехоплення

  • Зловмисник у тому ж мережевому сегменті (LAN, VLAN)
  • Відсутність шифрування (plain SIP, plain RTP)
  • Незахищений WiFi (eavesdropping у корпоративній WiFi)
  • Компрометований маршрутизатор/комутатор
  • BGP hijacking / DNS hijacking (великий масштаб)

📡 Перехоплення SIP-сигналізації

;; SIP REGISTER — перехоплений (Wireshark)
REGISTER sip:pbx.company.com SIP/2.0
Via: SIP/2.0/UDP 192.168.1.100:5060
From: <sip:alice@company.com>
To: <sip:alice@company.com>
Authorization: Digest
  username="alice",
  realm="pbx.company.com",
  nonce="abc123",
  uri="sip:pbx.company.com",
  response="d8b9f3a4..."

;; Зловмисник отримує:
;;  - логін: alice
;;  - realm: pbx.company.com
;;  - nonce + response → offline crack MD5

Інструменти перехоплення SIP

  • Wireshark + SIP dissector + RTP playback
  • sngrep — консольний SIP-аналізатор
  • tcpdump → PCAP → аналіз
  • SIPp — record mode
SIP digest (MD5) — не є стійким шифруванням. Зловмисник може виконати офлайн-атаку на перехоплений Authorization заголовок за допомогою hashcat або sipcrack.

🎵 Перехоплення RTP медіа-потоків

Alice ──── RTP (UDP, plain G.711) ────► PBX/Gateway ────► Bob
                    │
            Зловмисник (ARP spoof / port mirror)
                    │
            Wireshark → "Telephony → RTP → Play Streams"
                    │
            Прослуховує розмову у реальному часі
            або зберігає WAV-файл

Технічна деталь

  • RTP без SRTP = голос у відкритому тексті
  • Wireshark автоматично збирає та відтворює RTP-потоки
  • SDP містить порти → зловмисник точно знає куди слухати
  • G.711 (PCMU/PCMA) — найпростіший для відтворення, без розкодування

Перехоплення DTMF (PIN-коди)

  • DTMF in-band (аудіо) — відтворюється разом із голосом
  • RFC 2833 / RFC 4733 — окремі RTP події, легко парсити
  • SIP INFO — DTMF у SIP-повідомленні → перехоплюється разом із SIP
  • Загроза: PIN банківського IVR, паролі автосекретаря

🕵️ Man-in-the-Middle (MitM) атаки

MitM у VoIP — зловмисник вставляє себе між двома учасниками комунікації, перехоплюючи та/або модифікуючи сигналізацію і медіа.

Методи встановлення MitM

  • ARP Spoofing — підміна MAC у локальній мережі
  • DNS Spoofing — підміна A/SRV записів SIP-домену
  • DHCP Spoofing — видача підробленого default gateway
  • Rogue SIP Proxy — підроблений проксі у локальній мережі
  • BGP hijacking — перехоплення трафіку на рівні AS
  • SSL stripping — зниження TLS до plain SIP

Що може зробити зловмисник

  • Прослуховування — перехоплення SIP+RTP
  • Модифікація — зміна SDP (підміна RTP-портів)
  • Re-INVITE атака — перенаправлення медіа на власний хост
  • Call hijacking — захоплення активного дзвінку
  • Toll fraud — зміна номера в INVITE
  • Ін'єкція голосу — вставка аудіо у розмову

🔀 ARP Spoofing у VoIP-мережі

Нормальний стан:
  Alice (192.168.1.10) ──────────────────── PBX (192.168.1.1)
      MAC: AA:BB:CC:DD:EE:01              MAC: 11:22:33:44:55:01

Після ARP Spoofing:
  Alice              Зловмисник (192.168.1.99)           PBX
  │                  MAC: FF:FF:FF:FF:FF:FF               │
  │ Надсилає трафік  │ "Я — PBX"   "Я — Alice"            │
  │─────────────────►│◄──────────────────────────────────►│
  │                  │                                     │
  │  SIP INVITE ──►  │  Перехоплює, читає,      ──► INVITE│
  │  RTP ──────────► │  ретранслює (або модифікує)── RTP ►│
Інструменти: arpspoof (dsniff), Ettercap, Bettercap, scapy. Виявлення: arping, dynamic ARP inspection (DAI) на комутаторах.
Захист: Dynamic ARP Inspection (DAI) на L2 комутаторах, 802.1X автентифікація портів, VLAN сегрегація VoIP трафіку

🔐 Захист медіа: SRTP та ZRTP

SRTP — Secure RTP (RFC 3711)

  • Шифрування: AES-128/256-CTR
  • Автентифікація: HMAC-SHA1
  • Захист від replay-атак (sequence number)
  • Ключі погоджуються через SDES (у SDP) або DTLS-SRTP
  • SDES передає ключ у SDP — вразливий якщо SIP не захищений
  • DTLS-SRTP (WebRTC) — ключі погоджуються безпосередньо між UA

ZRTP — Z Real-time Transport Protocol (RFC 6189)

  • Автор: Філіп Циммерман (творець PGP)
  • Обмін ключами Diffie-Hellman у медіа-каналі
  • Не залежить від PKI та сигналізації
  • SAS (Short Authentication String) — голосова верифікація
  • Захист від MitM без сертифікатів
  • Використовується: Signal, Linphone, Twinkle
Правило: SRTP без TLS для сигналізації не вирішує проблему — ключ у SDP може бути перехоплений. Повний захист = TLS (SIP) + DTLS-SRTP (медіа) або ZRTP.

🔒 Захист сигналізації: TLS / SIPS

SIP over TLS (SIPS)

  • Порт 5061/TCP
  • URI: sips:alice@atlanta.com
  • Шифрування: TLS 1.2 / TLS 1.3
  • Автентифікація сервера через X.509
  • Захищає SIP-заголовки, SDP, credentials
  • Запобігає перехопленню та модифікації
TLS не захищає hop-by-hop якщо проміжні proxy в мережі не використовують TLS між собою.

Типові помилки конфігурації TLS

❌ Часті вразливості

  • Самопідписані сертифікати без перевірки
  • TLS 1.0/1.1 — вразливі (POODLE, BEAST)
  • Weak cipher suites (RC4, 3DES)
  • Відключена перевірка сертифіката UA
  • Змішані TLS/non-TLS hop на шляху
Рекомендація: TLS 1.3 + ECDSA/RSA-2048+ + перевірка CN/SAN + HSTS для веб-UI

  • Ціль: корпоративні облікові дані через голос

🏰 Стратегія глибинного захисту VoIP

🌐 Периметр

  • SBC — єдина точка входу
  • Anti-DDoS (scrubbing)
  • Firewall: SIP ALG off, stateful
  • Geo-blocking підозрілих AS
  • BCP38 anti-spoofing

🖧 Мережа

  • VLAN сегрегація VoIP
  • 802.1X аутентифікація портів
  • DAI (Dynamic ARP Inspection)
  • DHCP snooping
  • QoS / DSCP маркування

📱 Застосунок

  • TLS/SIPS + SRTP скрізь
  • Сильна автентифікація SIP
  • Fail2ban + rate limiting
  • CDR-моніторинг у реальному часі
  • Регулярний патчинг PBX
Принцип: жодний окремий засіб захисту не є достатнім. Ефективна безпека VoIP = багаторівневий захист (мережа + протокол + застосунок + організація + моніторинг).
