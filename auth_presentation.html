⚠️ Загрози автентифікації у VoIP

Сигналізація VoIP передається відкритими мережами. Без надійної автентифікації зловмисник може видавати себе за законного учасника, перехоплювати виклики та обходити тарифікацію.

Основні вектори атак

  • Реєстраційне викрадення — перехоплення реєстрації SIP
  • Toll Fraud — несанкціонована тарифікація
  • Caller ID Spoofing — підміна номера
  • MITM — підміна SIP-проксі
  • Replay Attack — повторне відтворення повідомлень
  • Brute Force — перебір SIP-паролів
  • Unauthorized Call Forwarding

Наслідки компрометації

  • Прослуховування переговорів
  • Фінансові збитки (toll fraud)
  • Порушення конфіденційності
  • Репутаційні ризики (спуфінг ідентичності)
  • Відмова в обслуговуванні (DoS)
  • Витік корпоративних даних
  • Правові наслідки (GDPR, закони про перехоплення)

🔐 Моделі автентифікації: огляд

🔑 Знання (Something you know)

  • Пароль / PIN
  • SIP Digest (MD5 / SHA-256)
  • Shared secret

🪪 Власність (Something you have)

  • X.509 сертифікат
  • SIM-картка
  • OAuth токен
  • Hardware token / FIDO2

🌐 Контекст (Something you are)

  • IP-адреса (мережевий контроль)
  • Біометрія (голос)
  • P-Asserted-Identity (оператор)
У VoIP найширше застосовуються: SIP Digest (пароль), TLS + сертифікати (PKI) і STIR/SHAKEN (ланцюг довіри оператора). OAuth 2.0 — новий стандарт для федеративних сценаріїв.

🔏 SIP Digest-автентифікація (RFC 3261)

SIP запозичив механізм HTTP Digest Authentication (RFC 7616). Пароль ніколи не передається у відкритому вигляді — передається лише хеш-відповідь.

Схема виклику-відповіді

UAC                   SIP Server
 |                        |
 |--- REGISTER ---------->|
 |                        |
 |<-- 401 Unauthorized ---|
 |    WWW-Authenticate:   |
 |    Digest realm=...,   |
 |    nonce="abc123",     |
 |    algorithm=MD5       |
 |                        |
 |--- REGISTER ---------->|
 |    Authorization:      |
 |    Digest username=.., |
 |    response="hash"     |
 |                        |
 |<-- 200 OK -------------|

Параметри WWW-Authenticate

  • realm — домен автентифікації
  • nonce — одноразове значення сервера
  • algorithm — MD5 або SHA-256
  • qop — якість захисту (auth / auth-int)
  • opaque — непрозорі дані сервера
  • stale — ознака застарілого nonce
Увага: MD5 офіційно вилучено з RFC 7616. Рекомендується SHA-256.

🧮 Розрахунок Digest-відповіді

Алгоритм (qop=auth)

HA1 = MD5(username:realm:password)

HA2 = MD5(method:digestURI)

response = MD5(
  HA1 : nonce : nc :
  cnonce : qop : HA2
)
cnonce — клієнтський nonce (захист від replay)
nc — лічильник запитів (nonce count)

Приклад Authorization-заголовка

Authorization: Digest
  username="alice",
  realm="atlanta.example.com",
  nonce="84a4cc6f3082121f32b42a2187831a9e",
  uri="sip:atlanta.example.com",
  qop=auth,
  nc=00000001,
  cnonce="0a4f113b",
  response="6629fae49393a05397450978507c4ef1",
  algorithm=MD5
Обмеження: MD5-хеш уразливий до офлайн-атак перебором. Перехоплений 401+REGISTER дозволяє зловмиснику відновити пароль без взаємодії з сервером.

💀 Обмеження Digest та атаки

Слабкі місця Digest

  • Не забезпечує конфіденційність сигналізації
  • Не захищає від MITM (без TLS)
  • MD5: хмарний перебір за секунди
  • Одностороння автентифікація (клієнт → сервер)
  • Відсутній захист цілісності заголовків SIP
  • Nonce-флудинг — DoS-вектор

Відомі атаки

  • SIPVicious / svcrack — сканування та злом SIP-паролів
  • Hashcat / John the Ripper — офлайн перебір MD5
  • Nonce replay — повторне використання nonce
  • Downgrade attack — примус до MD5 замість SHA-256
  • Registration hijacking — підміна Contact-заголовка
Реальний приклад: злом SIP-акаунту корпоративного PBX → toll fraud на $50 000 за вихідні.

🔒 TLS для захисту SIP-сигналізації

TLS (Transport Layer Security) шифрує SIP-трафік на транспортному рівні, захищаючи від прослуховування та MITM. SIP over TLS = SIPS.

Характеристики SIP/TLS

  • Порт 5061/TCP (vs 5060 для UDP/TCP)
  • Двостороння автентифікація (mutual TLS)
  • TLS 1.2 / TLS 1.3 (RFC 8446)
  • Шифрування: AES-128-GCM, ChaCha20-Poly1305
  • Захист цілісності та конфіденційності
  • X.509-сертифікати для ідентифікації

Модель довіри TLS у SIP

UAC ──[TLS]──► Proxy A ──[TLS]──► Proxy B ──[TLS]──► UAS
      ↑                                               ↑
 cert verify                                    cert verify

Hop-by-hop: кожен сегмент шифрується окремо
End-to-end: потрібен S/MIME або SRTP
TLS захищає лише hop-by-hop. Проміжні проксі бачать SIP-заголовки у відкритому вигляді.

🔗 SIPS URI та безпечна маршрутизація

Синтаксис та семантика

SIP URI:  sip:alice@atlanta.com
SIPS URI: sips:alice@atlanta.com
            ↑
     Вимагає TLS на всьому шляху
  • SIPS вимагає TLS на кожному hop
  • Проксі зобов'язаний відхилити виклик, якщо TLS недоступний
  • Поширюється на Contact-, Record-Route- та Route-заголовки
  • Використовується у Request-URI та To/From-заголовках

Алгоритм перевірки маршруту

  • Проксі отримує INVITE з SIPS Request-URI
  • Знаходить наступний hop через DNS NAPTR/SRV
  • Переконується, що запис вказує на TLS-транспорт
  • Якщо TLS недоступний → 502 Bad Gateway
  • Зберігає SIPS у Record-Route
RFC 3261 §19.1: SIPS — це зобов'язання забезпечити end-to-end захист сигналізаційного шляху.

✉️ S/MIME у SIP (RFC 3261 §23)

S/MIME забезпечує end-to-end захист тіла SIP-повідомлення (SDP, MIME-контент). На відміну від TLS, S/MIME шифрує тіло наскрізно — проміжні проксі не можуть його прочитати.

Можливості S/MIME

  • Підпис — цілісність і автентифікація відправника
  • Шифрування — конфіденційність SDP
  • Обидва разом — максимальний захист
  • Content-Type: application/pkcs7-mime
  • Content-Type: multipart/signed

Обмеження S/MIME у SIP

  • Потребує PKI-інфраструктури (CA, сертифікати)
  • Ускладнює роботу проміжних B2BUA
  • Шифрування SDP ускладнює медіа NAT traversal
  • Слабка підтримка у телефонних клієнтах
  • На практиці рідко застосовується
S/MIME у SIP — теоретично ідеальний, практично рідко реалізований підхід.

🎙️ SRTP — захист медіапотоків (RFC 3711)

SRTP (Secure RTP) — розширення RTP з шифруванням, автентифікацією та захистом від відтворення медіапотоків. Узгоджується через SDP m=audio ... RTP/SAVP.

Захисні механізми SRTP

  • Шифрування: AES Counter Mode (128/256 біт)
  • Автентифікація: HMAC-SHA1 (80-бітний тег)
  • Replay Window: захист від повторного відтворення
  • MKI (Master Key Identifier): ротація ключів
  • Мінімальний накладний трафік (~10 байт/пакет)

Управління ключами SRTP

  • SDES (RFC 4568): ключі у SDP в Base64 — потрібен TLS!
  • ZRTP (RFC 6189): DH-обмін у медіаплощині
  • DTLS-SRTP (RFC 5763): TLS поверх UDP для WebRTC
  • MIKEY (RFC 3830): мультимедійний обмін ключами
SDES без TLS — критична вразливість: ключі SRTP передаються у відкритому вигляді в SDP.

🔑 ZRTP — обмін ключами (RFC 6189)

ZRTP — протокол Філіпа Ціммермана (автора PGP) для узгодження ключів SRTP безпосередньо у медіаплощині без залежності від PKI чи SIP-сигналізації.

Принцип роботи ZRTP

Alice               Bob
  |--- Hello -------->|
  |<-- HelloACK ------|
  |--- Commit ------->|
  |<-- DHPart1 -------|
  |--- DHPart2 ------>|
  |<-- Confirm1 ------|
  |--- Confirm2 ----->|
  |<-- Conf2ACK ------|
  |  [SRTP активовано]|
  |   SAS: "zorbix"   | ← вербальна верифікація

Ключові особливості

  • DH (Diffie-Hellman) — Perfect Forward Secrecy
  • SAS (Short Authentication String) — захист від MITM: користувачі вголос порівнюють 4-буквений рядок
  • Не потребує PKI та сертифікатів
  • Зберігає rs1/rs2 для верифікації майбутніх дзвінків
  • Підтримується Linphone, Jitsi, csipsimple
ZRTP забезпечує справжній end-to-end захист без довіри до SIP-сервера.

🌐 DTLS-SRTP (RFC 5763) та WebRTC

DTLS-SRTP

  • DTLS = TLS поверх UDP (RFC 6347)
  • Handshake виконується у медіаплощині (порт RTP)
  • SDP містить відбиток сертифіката (a=fingerprint)
  • Генеруються майстер-ключі для SRTP
  • Захист: шифрування + автентифікація + PFS
SDP:
a=fingerprint:sha-256 4A:AD:B9:B1:...
a=setup:actpass
m=audio 49170 UDP/TLS/RTP/SAVPF 0

WebRTC та безпека

  • WebRTC обов'язково використовує DTLS-SRTP
  • Сигналізація через WebSocket + TLS
  • ICE для NAT traversal (STUN/TURN)
  • Браузери перевіряють відбитки сертифікатів
RFC 8827: WebRTC Security Architecture вимагає DTLS 1.2+ та забороняє SDES без TLS.
DTLS-SRTP — стандарт de facto для сучасних браузерних VoIP-рішень.

🛡️ H.235 — безпека в H.323

H.235 — стандарт ITU-T для забезпечення автентифікації, цілісності та конфіденційності в H.323-мережах. Визначає набір профілів безпеки.

Профілі H.235

ПрофільПризначення
H.235.1Базова автентифікація (пароль)
H.235.2Підпис (сигнатура), PKI
H.235.3Гібридна безпека (пароль + PKI)
H.235.6Голосове шифрування (AES)
H.235.7TLS/IPSec транспорт
H.235.8SRTP ключовий матеріал

H.235.1 — автентифікація паролем

  • Використовує ClearToken та CryptoH323Token
  • Вбудовується у RAS та H.225-повідомлення
  • HMAC-SHA1 або MD5 хеш паролю
  • Timestamp-захист від replay-атак
Більшість реалізацій H.323 підтримують лише H.235.1 (пароль). Повна PKI-інфраструктура (H.235.2) зустрічається рідко.

🪪 Стверджена ідентичність: P-Asserted-Identity

P-Asserted-Identity (PAI) (RFC 3325) — заголовок SIP, яким довірений SIP-проксі стверджує автентифіковану ідентичність абонента. Використовується в мережах операторів.

Схема роботи PAI

UAC ──REGISTER──► Proxy (Trust Domain)
                    │ автентифікація Digest/TLS
                    │ перевірка ідентичності
                    ▼
               INVITE з:
  P-Asserted-Identity: "Alice" <sip:alice@carrier.ua>
  P-Preferred-Identity: <sip:alice@example.com>
                    │
                    ▼
              Зовнішня мережа

Privacy та обмеження

  • Privacy: id — UAC просить приховати PAI від кінцевого отримувача
  • PAI передається лише між довіреними проксі
  • Видаляється на межі довіреної зони
  • Не забезпечує end-to-end верифікацію
  • Вразливість: зловмисний проксі може вставити довільний PAI
PAI — рішення для мереж одного оператора. Для міжмережевої верифікації потрібен STIR/SHAKEN.

📜 STIR — Secure Telephony Identity Revisited

STIR (RFC 8224, RFC 8225) — стандарт криптографічного підпису SIP-ідентичності на основі JWT-токенів та PKI. Мета: боротьба з caller ID spoofing.

Архітектура STIR

Originating SP          Terminating SP
    │                         │
    │  INVITE                 │
    │  Identity: eyJ...       │  ← PASSporT (JWT)
    │  ──────────────────────►│
    │                    перевірка підпису
    │                    через HTTPS (cert URL)
    │                    ОК → довірений
    │                    FAIL → підозрілий

PASSporT — структура токена

Header:
{ "alg": "ES256", "typ": "passport",
  "ppt": "shaken", "x5u": "https://cert.ca/cert.pem" }

Payload:
{ "orig": {"tn": "380441234567"},
  "dest": {"tn": ["380679876543"]},
  "iat": 1711880000,
  "attest": "A" }
attest: A = повна атестація (CP авторизований для номера), B = часткова, C = маршрутна

🤝 SHAKEN — розгортання STIR у операторів

SHAKEN (Signature-based Handling of Asserted information using toKENs) — операційна система для розгортання STIR у телефонних мережах. Розроблена ATIS/SIP Forum, обов'язкова у США (TRACED Act, 2021).

Компоненти SHAKEN

  • STI-CA — Certification Authority для SP-сертифікатів
  • STI-PA — Policy Administrator (ATIS у США)
  • STI-AS — Authentication Service (підписує виклики)
  • STI-VS — Verification Service (перевіряє підписи)
  • STI-CR — Certificate Repository (HTTPS)

Рівні атестації та відображення

РівеньЗначенняВідображення
AПовна атестація✅ Підтверджений
BЧасткова⚠️ Частково
CМаршрутна❓ Невідомий
—Без підпису🚫 Підозрілий
В Україні STIR/SHAKEN перебуває на етапі обговорення в НКРЗІ.

🎫 OAuth 2.0 у SIP (RFC 8898)

RFC 8898 стандартизує використання OAuth 2.0 токенів (JWT Bearer) для автентифікації SIP-агентів. Забезпечує федерацію та делеговану авторизацію.

Схема потоку OAuth у SIP

UAC ──────────────────────► Authorization Server
                              (Keycloak, Azure AD...)
UAC ◄── access_token (JWT) ──

UAC ──── REGISTER ──────────► SIP Registrar
         Authorization:
         Bearer eyJ...

SIP Registrar ─── introspect/JWKS ──► AS
SIP Registrar ◄── valid ───────────

SIP Registrar ──── 200 OK ─────────► UAC

Переваги OAuth у SIP

  • Централізоване управління ідентичністю (SSO)
  • Коротко-живі токени (exp claim)
  • Scopes для тонкого контролю доступу
  • Підтримка MFA на рівні AS
  • Інтеграція з корпоративним IdP
Microsoft Teams, Cisco Webex та RingCentral використовують OAuth/OIDC для автентифікації SIP-клієнтів.

🕵️ Атака: Реєстраційне викрадення

Registration hijacking — зловмисник надсилає REGISTER із легітимним From/To, але підміненим Contact, перенаправляючи вхідні виклики на свій UA.

Вектори атаки

Легітимний REGISTER:
From: <sip:alice@example.com>
Contact: <sip:alice@192.168.1.10>
Expires: 3600

Атака: перехоплення/підробка
From: <sip:alice@example.com>
Contact: <sip:attacker@10.0.0.5>
Expires: 3600
Authorization: Digest ... (злом)

Заходи захисту

  • TLS — захист від перехоплення REGISTER
  • SIP AOR binding — прив'язка до IP/мережі
  • Аномалія-детекція — зміна Contact з нового IP
  • Max-Forwards обмеження
  • Short registration interval — зменшення вікна атаки
  • Admin сповіщення при зміні Contact
  • RFC 5626 — SIP Outbound (постійне з'єднання)
Компрометація Digest + MITM → повний захоплення облікового запису SIP.

🔁 Атаки: Replay та Brute Force

Replay Attack

  • Зловмисник перехоплює автентифіковане повідомлення
  • Повторно відправляє його пізніше
  • Захисти від replay у SIP Digest:
    • nonce — одноразове значення сервера
    • nc — лічильник, зростає монотонно
    • cnonce — клієнтський nonce
  • Якщо nonce не валідується → replay проходить
  • H.235: timestamp у токені (±5 хвилин)

Brute Force на SIP Digest

  • Перехоплений 401 + повторний REGISTER → offline атака
  • Hashcat: -m 11400 (SIP Digest MD5)
  • GPU: ~1 млрд MD5/с → слабкий пароль за секунди
  • SIPVicious svcrack: online перебір
Захист:
• Мінімум 12+ символів, складні паролі
• Account lockout після N спроб
• Fail2Ban / SBC rate limiting
• TLS (ускладнює перехоплення для offline атаки)
• SHA-256 замість MD5

🕸️ Атака: MITM на сигналізацію

Схема MITM-атаки

Alice                  Attacker              SIP Server

Вектори та захист

  • ARP Spoofing + SIP intercept — в локальній мережі
  • Rogue SIP Proxy — підроблений проксі
  • DNS Spoofing — підміна SRV-запису SIP
Захист:
• TLS + верифікація сертифіката (mutual TLS)
• HSTS-аналог для SIP: SIPS URI
• ZRTP SAS верифікація (end-to-end)
• SRTP — навіть при MITM сигналізації медіа зашифровано
• SBC з валідацією топології

🚧 Session Border Controller (SBC)

SBC — прикордонний елемент VoIP-мережі, що виконує функції security gateway, NAT traversal, topology hiding та policy enforcement для SIP-трафіку.

Функції безпеки SBC

  • TLS termination — розшифрування/перешифрування SIP
  • Topology hiding — приховування внутрішньої топології
  • DoS/DDoS protection — rate limiting, blacklisting
  • SIP anomaly detection — аномальні повідомлення
  • Toll fraud prevention — обмеження на напрямки
  • SRTP relay — перешифрування медіа
  • Regulatory compliance — CALEA/SORM

🔐 IPSec для VoIP-інфраструктури

IPSec може захищати VoIP на мережевому рівні, особливо у сегментах між операторськими вузлами (SBC ↔ SoftSwitch, SBC ↔ Media Gateway).

Режими IPSec у VoIP

  • Transport mode — між двома хостами (захист лише корисного навантаження)
  • Tunnel mode — VPN між майданчиками (захист усього пакета)
  • IKEv2 (RFC 7296) — обмін ключами
  • ESP — шифрування + автентифікація
  • AH — лише автентифікація (не рекомендується для NAT)

Переваги та обмеження

IPSec між UA та проксі потребує ESP NAT-T (RFC 3948) для роботи через NAT.

📋 PKI та управління сертифікатами

PKI є фундаментом для TLS, STIR/SHAKEN, S/MIME та mTLS у VoIP-мережах. Правильне управління сертифікатами критично для безпеки.

Типи сертифікатів у VoIP

  • SIP-сервер: CN = FQDN проксі (напр. sip.example.com)
  • SIP UA: SAN = SIP URI (sip:alice@example.com) (RFC 5922)
  • STIR: TN Authorization List (RFC 8226)
  • WebRTC: самопідписані (перевірка по fingerprint)

Критичні вимоги

  • Перевіряти CN/SAN і ланцюг CA
  • OCSP/CRL для перевірки статусу
  • Автоматична ротація (ACME / Let's Encrypt)

✅ Практичні рекомендації та чеклист

🔒 Сигналізація

  • TLS 1.2+ на всіх SIP-з'єднаннях
  • SHA-256 замість MD5 у Digest
  • Mutual TLS (mTLS) між серверами
  • SIPS URI для чутливих потоків
  • Перевірка CN/SAN сертифікатів
  • Вимкнути незашифровані UDP/TCP назовні

🎙️ Медіа

  • SRTP обов'язково (заборонити RTP)
  • DTLS-SRTP для WebRTC
  • SDES лише з TLS-захистом SDP
  • ZRTP для кінцевих точок high-sec
  • AES-256-GCM де підтримується
  • Заборонити SDES без TLS

🛡️ Інфраструктура

  • SBC на межі мережі
  • Fail2Ban / rate limiting
  • Сильні паролі (12+, складні)
  • Account lockout після 5 спроб
  • STIR/SHAKEN для міжмережевих дзвінків
  • OAuth 2.0 для enterprise SSO
  • Audit log усіх REGISTER / INVITE
  • Регулярний аудит безпеки

📝 Підсумок лекції

Ключові механізми автентифікації

  • SIP Digest — базовий рівень (SHA-256, qop=auth)
  • TLS / mTLS — транспортна безпека та PKI-автентифікація
  • STIR/SHAKEN — верифікація ідентичності між операторами
  • OAuth 2.0 — федеративна автентифікація в enterprise
  • H.235 — профілі безпеки для H.323-мереж

Ключові механізми захисту сигналізації

  • SIPS URI — примусовий TLS по всьому маршруту
  • SRTP + DTLS — захист медіапотоків
  • ZRTP — end-to-end без PKI
  • SBC — граничний захист та DoS-митигація
  • IPSec — мережевий рівень між операторськими вузлами
Головний принцип: Defense-in-depth — жоден одиночний механізм не забезпечує повний захист. Необхідно поєднувати автентифікацію, шифрування сигналізації, шифрування медіа та моніторинг.
