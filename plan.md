# План лекції 7: Безпека IP-АТС на прикладі Asterisk

**Курс:** Технології VoIP  
**Аудиторія:** Студенти магістратури, напрям «Кібербезпека»  
**Тривалість:** 2 академічні години (90 хвилин)  
**Форма:** Лекція з демонстраціями та практичними прикладами конфігурацій

---

## 1. Навчальні цілі

Після лекції студенти зможуть:

1. **Пояснити фундаментальні принципи безпеки** (CIA, AAA, Saltzer & Schroeder) та показати їх відображення у захисті IP-АТС.
2. Пояснити архітектуру захищеної IP-АТС та роль принципу мінімізації привілеїв.
3. Налаштовувати ACL на рівні IP у конфігурації Asterisk та на мережевому рівні (iptables/nftables, fail2ban).
4. Використовувати контексти `chan_sip` / `chan_pjsip` для ізоляції груп користувачів у диалплані.
5. Виявляти та усувати типові вразливості диалплану (Toll Fraud, DISA, некоректна валідація введення).
6. Застосовувати комплекс захисних заходів: TLS/SRTP, Digest Auth, захист AMI, аудит конфігурацій.
7. **Обґрунтовувати** кожне рішення щодо безпеки посиланням на конкретний принцип ІБ.

### Блок 2. Теоретичні принципи безпеки (Слайди 5–10) — ~15 хв

> **Мета блоку:** дати студентам теоретичний фундамент, на який спираються всі наступні практичні рішення. Кожен принцип завершується прикладом з Asterisk.

**Слайд 5. CIA Тріада в контексті IP-АТС**
- **Confidentiality:** несанкціонований перехват SIP/RTP → загроза комерційній таємниці.
  - *Asterisk:* TLS для сигналізації + SRTP для медіа.
- **Integrity:** підміна SIP-повідомлень, Caller ID spoofing → шахрайські виклики.
  - *Asterisk:* SIP Digest Auth + TLS з перевіркою сертифіката.
- **Availability:** DoS/flood на SIP-порт → АТС недоступна, бізнес зупинено.
  - *Asterisk:* fail2ban + rate limiting + VLAN-ізоляція.
- Висновок: захист IP-АТС = захист усіх трьох властивостей одночасно; атаки зазвичай порушують декілька.

**Слайд 6. AAA Framework**
- **Authentication** («Хто ти?»): SIP Digest Auth, ACL по IP, TLS-сертифікат.
- **Authorization** («Що можна?»): контекст у `extensions.conf`, ACL permit/deny, extension patterns.
- **Accounting** («Що робив?»): CDR / CEL, `/var/log/asterisk/`, SIEM-інтеграція.
- Типові вразливості: слабка Authentication відкриває Authorization; відсутній Accounting → інцидент не виявлено.

**Слайд 7. 8 Принципів Saltzer & Schroeder (1975)**
- Класична робота «The Protection of Information in Computer Systems».
- Таблиця: кожен принцип → реалізація в Asterisk:
  1. Economy of Mechanism → мінімальний набір модулів (`autoload=no`).
  2. Fail-Safe Defaults → `allowguest=no`, `deny=0.0.0.0/0` першим.
  3. Complete Mediation → ACL на кожному endpoint; кожен виклик через контекст.
  4. Open Design → Asterisk open-source; безпека через конфіг, не обфускацію.
  5. Separation of Privilege → окремі контексти, окремі AMI-акаунти.
  6. Least Privilege → запуск від `asterisk`; обмежені права AMI.
  7. Least Common Mechanism → VLAN, ізоляція процесів, не shared контексти.
  8. Psychological Acceptability → іменовані контексти; коментарі у конфігах.

**Слайд 8. Fail-Safe Defaults: безпечна відмова**
- Теорія: система за замовчуванням **забороняє** все, дозволяє лише явно налаштоване; якщо перевірка неможлива — відмовити у доступі.
- Fail-Open (небезпечний) vs Fail-Secure (правильний): порівняння патернів.
- Asterisk: `allowguest=no`, `deny=0.0.0.0/0` першим, `alwaysauthreject=yes`.
- Наслідок порушення: `allowguest=yes` + відсутність ACL = Toll Fraud без злому паролів.

**Слайд 9. Мінімізація поверхні атаки**
- Теорія: Attack Surface — сукупність усіх точок входу. Принцип: зменшити кількість і розмір до мінімуму.
- Повна vs мінімізована поверхня атаки Asterisk: порти, модулі, API.
- Вимкнення невикористаних модулів (`modules.conf`: `autoload=no`, явний список).
- Обмеження RTP-діапазону (`rtp.conf`: `rtpstart`/`rtpend`), вимкнення HTTP/ARI якщо не потрібен.

**Слайд 10. Zero Trust для IP-АТС**
- Теорія: «Ніколи не довіряй, завжди перевіряй» — жоден компонент не є автоматично довіреним, навіть у внутрішній мережі.
- Традиційна модель (периметр) vs Zero Trust: схема порівняння.
- Zero Trust в Asterisk: Digest Auth для КОЖНОГО endpoint (не лише зовнішніх), мікросегментація контекстів.
- Чому це важливо: компромет одного внутрішнього пристрою не дає доступу до всього.

### Блок 3. Принцип мінімізації привілеїв (Слайди 11–14) — ~12 хв

**Слайд 11. Principle of Least Privilege (PoLP) → Asterisk**
- Поглиблення принципу #6 Saltzer & Schroeder у контексті IP-АТС.
- Чотири рівні PoLP: ОС → мережа → Asterisk ACL → диалплан.
- Зв'язок з Defense-in-Depth: кожен рівень PoLP = окремий рубіж.

**Слайд 12. Захист на рівні операційної системи**
- Запуск Asterisk від непривілейованого користувача (`asterisk:asterisk`).
- Параметри `-U` / `-G` при старті демона.
- Права доступу до `/etc/asterisk/`, `/var/lib/asterisk/`, `/var/log/asterisk/`.
- `systemd` unit-файл: `NoNewPrivileges`, `ProtectSystem`, `PrivateTmp`.

**Слайд 13. Захист конфігураційних файлів**
- Відокремлення секретів: паролі SIP, AMI credentials.
- `#include` для зберігання секретів у окремих файлах з жорсткими правами.
- Шифрування диску / Vault для sensitive даних.
- Версійний контроль конфігурацій (Git + pre-commit hooks для виявлення секретів).

**Слайд 14. Мережева ізоляція АТС**
- Розміщення Asterisk у виділеному VLAN.
- Схема: LAN Users VLAN ↔ PBX VLAN ↔ SIP Trunk VLAN.
- Принцип: кожен компонент має доступ тільки до необхідних ресурсів.

### Блок 4. ACL — Контроль доступу на рівні IP (Слайди 15–19) — ~15 хв

**Слайд 15. ACL: концепція та рівні (реалізація Complete Mediation)**
- Теоретичне обґрунтування: ACL реалізує принцип #3 Saltzer & Schroeder — Complete Mediation.
- Рівень 1: Мережевий файрвол (iptables / nftables / pfSense).
- Рівень 2: Asterisk внутрішні ACL (`permit`/`deny` у конфігах).
- Рівень 3: fail2ban / Snort — динамічний захист.
- Defense-in-depth: жоден рівень не є достатнім сам по собі.

**Слайд 16. ACL у chan_sip (sip.conf)**
- Директиви `permit`, `deny`, `acl`.
- Приклади конфігурації: дозвіл тільки корпоративної підмережі.
- `contactpermit`, `contactdeny` — перевірка Contact-header у REGISTER.
- `allowsubscribe`, `allowoverlap` — зменшення поверхні атаки.

**Слайд 17. ACL у chan_pjsip (pjsip.conf)**
- Об'єкт `[acl]` типу `acl`.
- Прив'язка ACL до endpoint через `acl=` і `contact_acl=`.
- Порівняння підходів chan_sip і chan_pjsip.

**Слайд 18. ACL для AMI та HTTP**
- `manager.conf`: `permit`, `deny` для AMI-підключень.
- `http.conf`: обмеження доступу до embedded HTTP-сервера.
- Принцип: AMI відкритий тільки для localhost / моніторинг-хостів (PoLP + Attack Surface).

**Слайд 19. fail2ban + iptables/nftables**
- Як fail2ban читає лог Asterisk і блокує IP-адреси.
- Конфігурація `asterisk.conf` jail та filter.
- `nftables` правила для обмеження SIP-реєстрацій (rate limiting).
- Команда: `nft add rule inet filter input ip saddr ...`.

### Блок 5. Контексти та ізоляція груп користувачів (Слайди 19–22) — ~13 хв

**Слайд 19. Контексти у Asterisk: реалізація принципу Separation of Privilege**
- Теоретичне обґрунтування: контексти реалізують принцип #5 Saltzer & Schroeder.
- Контекст як простір імен для extension'ів у диалплані.
- Зв'язок: SIP endpoint → context → доступні extension.
- Схема: внутрішні користувачі, зовнішній trunk, IVR — у різних контекстах.

**Слайд 20. Ізоляція груп у chan_sip**
- Параметр `context=` в `[general]`, `[peer]`, `[user]`.
- Приклад: `context=internal` для IP-телефонів, `context=from-trunk` для SIP-провайдера.
- Чому небезпечно використовувати один контекст для всіх.

**Слайд 21. Ізоляція груп у chan_pjsip**
- `context=` в endpoint-секції.
- Конфігурація trunk endpoint vs user endpoint.
- `identify` об'єкти для автентифікації по IP без пароля (і ризики).

**Слайд 22. Ієрархія контекстів: include**
- Директива `include => ` в extensions.conf.
- Дерево включень: від обмеженого до розширеного доступу.
- Типова помилка: `include => from-internal` у контексті trunk.
- Best practice: явний дозвіл, заборона за замовчуванням (Fail-Safe Default).

### Блок 6. Безпека диалплану (Слайди 23–29) — ~18 хв

**Слайд 23. Загрози на рівні диалплану**
- Toll Fraud: несанкціоновані дзвінки на міжнародні / преміум номери.
- DISA (Direct Inward System Access) — відкритий доступ до ліній.
- Voicemail → outbound: вихід у зовнішню мережу через VM-меню.
- Вразливості AGI/FastAGI-скриптів.

**Слайд 24. Toll Fraud: механізм і масштаб**
- Як зловмисник отримує доступ до SIP-акаунта.
- Схема: brute force → реєстрація → дзвінок на дорогі номери → збиток.
- Реальні цифри: збитки телекомпаній та підприємств.
- Типові цілі: +900, міжнародні, преміум-напрямки.

**Слайд 25. Захист від Toll Fraud у диалплані**
- Whitelist дозволених напрямків у `extensions.conf` (Fail-Safe Default).
- `CALLERID(num)` / `CHANNEL(peerip)` — перевірка джерела.
- Заборона `_+.`, `_011.`, `_1900NXXXXXX` паттернів для зовнішніх контекстів.
- `AMPUSER` / `AMAFLAGS` для CDR-аудиту (Accounting у AAA).

**Слайд 26. Валідація введення в диалплані**
- Небезпека довільного введення: `GotoIf`, `Exec`, `System()`.
- `SanitizeInput` через `FILTER()` функцію.
- Регулярні вирази у Extension паттернах: дозволяти тільки цифри.
- Обережно з `AGI()`, `System()`, `SHELL()` — можливе виконання команд.

**Слайд 27. Вразливості DISA**
- Що таке DISA і навіщо використовується.
- Чому DISA без PIN — критична вразливість (порушення Fail-Safe Defaults + AAA).
- Захист: складний PIN, обмеження часу, whitelist номерів, логування.
- Рекомендація: уникати DISA або замінити на VPN + внутрішній дзвінок.

**Слайд 28. Захист Voicemail та IVR**
- Типова вразливість: вихід у external через VM-опцію «передзвони».
- Заборона `OutgoingCallExten` або обмеження контекстом.
- IVR: обмеження кількості спроб введення, таймаути, логування.
- DTMF injection в IVR: валідація довжини та формату введення.

**Слайд 29. Ліміти та антифрод-механізми**
- `GROUP()` / `GROUP_COUNT()` — обмеження одночасних викликів на акаунт.
- `REALTIME` / CDR: виявлення аномальних паттернів.
- Зовнішні антифрод-системи: FraudBuster, ASTPP, власні AGI-скрипти.
- Алерти: SMS/email при перевищенні порогу витрат.

### Блок 7. Додаткові механізми захисту (Слайди 30–33) — ~10 хв

**Слайд 30. SIP Digest Authentication**
- Механізм challenge-response у Asterisk.
- `secret=` у peer-секції: вимоги до паролів (довжина, складність).
- `alwaysauthreject=yes` — захист від enumeration.
- `maxexpirey`, `minexpirey` — обмеження терміну реєстрацій.

**Слайд 31. TLS та SRTP в Asterisk**
- Конфігурація TLS для SIP у `sip.conf` / `pjsip.conf`.
- Генерація сертифікатів: Let's Encrypt або внутрішній CA.
- `encryption=yes` у PJSIP endpoint для SRTP.
- `media_encryption_optimistic` vs `required`.

**Слайд 32. Захист AMI (Asterisk Manager Interface)**
- AMI — потужний інтерфейс, небезпечний при неправильному налаштуванні.
- `manager.conf`: мінімальні права (`read`, `write`), обмеження по IP.
- Уникати plain-text; розглянути ARI (Asterisk REST Interface) з HTTPS.
- Принцип: кожна система моніторингу — окремий AMI-акаунт з мінімальними правами.

**Слайд 28. Логування, аудит і моніторинг**
- `logger.conf`: рівні `VERBOSE`, `SECURITY`, `WARNING`.
- SECURITY-events: реєстрація невдалих аутентифікацій.
- CDR (Call Detail Records) та CEL (Channel Event Logging).
- Інтеграція з SIEM (ELK Stack, Splunk, Graylog).
- Команди: `asterisk -rx "core show channels"`, `sngrep`, Wireshark.

### Методичні прийоми
- **Теорія → Практика:** кожен принцип безпеки спочатку пояснюється теоретично, потім демонструється на реальній конфігурації Asterisk.
- **Live demo:** показ конфігурацій у реальному Asterisk або Docker-контейнері.
- **Кейс-стаді:** реальний кейс Toll Fraud — аналіз логів, пошук вразливості, визначення порушеного принципу.
- **Pitfall-секції:** що не треба робити та чому — з посиланням на конкретний порушений принцип.
- **Checklists:** студенти заповнюють чек-ліст для умовної компанії та вказують, який принцип ІБ реалізує кожен пункт.

- Saltzer & Schroeder, *«The Protection of Information in Computer Systems»*, 1975 — фундаментальна стаття про принципи захисту
- NIST SP 800-53 — каталог засобів безпеки (Security and Privacy Controls)
- OWASP VoIP Security Project — практичні керівництва

## 6. Ключові терміни

| Термін | Визначення |
|--------|-----------|
| **CIA Тріада** | Confidentiality, Integrity, Availability — фундаментальна модель ІБ |
| **AAA** | Authentication, Authorization, Accounting — тріада контролю доступу |
| **Fail-Safe Default** | Принцип: система за замовчуванням забороняє все, дозволяє явно задане |
| **Attack Surface** | Сукупність точок входу, через які зловмисник може взаємодіяти з системою |
| **Zero Trust** | Архітектурний принцип: «Ніколи не довіряй, завжди перевіряй» |
| **Defense-in-Depth** | Ешелонована оборона: кілька незалежних рівнів захисту |
| **Saltzer & Schroeder** | 8 принципів безпечного проектування (1975): PoLP, Fail-Safe, Economy тощо |
| **ACL** | Access Control List — список правил permit/deny для IP-адрес |
| **Toll Fraud** | Несанкціоноване використання телефонних ліній для дорогих дзвінків |
| **Контекст** | Простір імен у диалплані Asterisk, що ізолює групи extension |
| **PoLP** | Principle of Least Privilege — мінімум необхідних прав |
| **DISA** | Direct Inward System Access — вхід у АТС ззовні з відкритим доступом до ліній |
| **AMI** | Asterisk Manager Interface — API для управління та моніторингу |
| **chan_pjsip** | Сучасний SIP-драйвер Asterisk на базі бібліотеки PJSIP |
| **chan_sip** | Застарілий (legacy) SIP-драйвер Asterisk |
| **CDR** | Call Detail Record — запис про дзвінок |
| **CEL** | Channel Event Logging — детальний журнал подій каналу |

## 7. Оцінювання знань

### Контрольні запитання (для самоперевірки)

**Теоретичні принципи:**
1. Поясніть CIA Тріаду та наведіть по одному прикладу загрози й захисту для кожного елементу в контексті IP-АТС.
2. Що таке AAA Framework? Яка різниця між Authentication та Authorization? Наведіть приклади з Asterisk.
3. Назвіть 5 з 8 принципів Saltzer & Schroeder та вкажіть їх реалізацію в Asterisk.
4. Що таке Fail-Safe Default? Наведіть два приклади «fail-open» і два «fail-secure» у конфігурації Asterisk.
5. Поясніть принцип Zero Trust та чому автентифікація внутрішніх IP-телефонів важлива навіть у захищеній мережі.

**Практичні питання:**
6. Що таке принцип мінімізації привілеїв і як він застосовується до конфігурації Asterisk на рівні ОС?
7. Яка різниця між `permit`/`deny` у `sip.conf` і правилами iptables?
8. Чому небезпечно мати один контекст для внутрішніх користувачів і зовнішнього SIP-транку?
9. Опишіть сценарій атаки Toll Fraud крок за кроком. Який принцип безпеки допомагає кожному захиснику?
10. Які Extension-паттерни слід заборонити для запобігання несанкціонованим дзвінкам?
11. Що таке DISA і чому вона є вразливістю без належного захисту?
12. Як fail2ban захищає Asterisk від брутфорс-атак?
13. Яка різниця між `media_encryption_optimistic` та обов'язковим SRTP?
