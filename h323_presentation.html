H.323 — перший зрілий стандарт для мультимедійних комунікацій в IP-мережах. Розуміння його архітектури є основою для аналізу безпеки VoIP-інфраструктур.

📡 Що таке H.323?

H.323 — стандарт ITU-T для мультимедійних комунікацій (голос, відео, дані) через пакетні мережі без гарантії якості обслуговування (наприклад, IP/Ethernet).

Ключові факти

  • Перша версія: 1996 рік (ITU-T H.323 v1)
  • Поточна редакція: H.323 v7 (2009)
  • Бінарний протокол на базі ASN.1
  • Пов'язаний стандарт: H.320 (ISDN), H.324 (PSTN)
  • Широко розповсюджений у 2000-х роках
  • Система відеоконференцій Cisco, Polycom, Tandberg

Сфери застосування

  • Корпоративні IP-АТС та відеоконференції
  • Шлюзи VoIP ↔ PSTN
  • Мережі операторів зв'язку (кінець 1990-х — 2010-х)
  • Системи дистанційного навчання (H.323 + H.239)
  • Відеоспостереження та телемедицина
  • Сумісність зі спадковим обладнанням

За 13 років (v1–v7) H.323 еволюціонував від базової LAN-відеосистеми до повноцінного корпоративного стандарту. Кожна версія закривала прогалини безпеки (H.235) та розширювала функціональність (H.460, H.239). Сьогодні v7 залишається останньою офіційною специфікацією ITU-T.

Компоненти стека

  • H.225 RAS — реєстрація, допуск, статус (UDP 1719)
  • H.225 Call Signaling — сигналізація Q.931 (TCP 1720)
  • H.245 — узгодження медіа-можливостей (TCP, динамічний порт)
  • RTP/RTCP — транспорт медіа (UDP, динамічні порти)
  • H.235 — безпека (інтегрована в H.225/H.245)
  • T.120 — протокол обміну даними (whiteboard, sharing)
Основна відмінність від SIP: H.323 використовує бінарне кодування ASN.1 (PER), а не текстовий формат.

🔢 ASN.1 та PER — бінарне кодування H.323

H.323 використовує ASN.1 (Abstract Syntax Notation One) — мову опису структур даних, та PER (Packed Encoding Rules) — правила їх компактного бінарного кодування. Це принципово відрізняє H.323 від текстових протоколів.

Чому ASN.1/PER?

  • Компактність: PER кодує без тегів і роздільників — мінімальний розмір пакету
  • Строга типізація: кожне поле має фіксований тип та обмеження
  • Стандартизована схема: автогенерація коду для будь-якої платформи
  • ALIGNED vs UNALIGNED PER: H.323 використовує ALIGNED (вирівнювання по байту)
Наслідок для безпеки: складний бінарний парсер = більша поверхня атаки. Помилки в парсингу ASN.1 — основна причина CVE у H.323-реалізаціях.

└──────┬──────┘                          │ RAS                       └──────┬──────┘
       │ RTP/RTCP                        │                                  │ RTP/RTCP
       │                    ┌────────────┴──────────┐                       │
       └────────────────────►   H.323 Gateway       ◄───────────────────────┘
                            │  (VoIP ↔ PSTN)        │
                            └────────────┬──────────┘

                            ┌────────────▼──────────┐
                            │   MCU (Multipoint     │

                            └───────────────────────┘
  • Terminal (TE) — кінцевий пристрій (IP-телефон, відеосистема, softphone)
  • Gatekeeper (GK) — центральний реєстратор зони, контроль доступу та маршрутизація
  • Gateway (GW) — міжмережевий шлюз (H.323 ↔ PSTN/H.320/SIP)
  • MCU — багатоточкова конференція (Multipoint Control Unit)
Архітектура H.323 є централізованою: Gatekeeper відіграє ключову роль у контролі доступу та адресації. Це відрізняє її від SIP, де відсутній обов'язковий центральний компонент. Така централізація спрощує управління, але створює єдину точку відмови.

🗺️ Концепція зони H.323

Зона H.323

  • Набір H.323-пристроїв, керованих одним Gatekeeper
  • Gatekeeper може бути обов'язковим (зареєстровані термінали) або необов'язковим
  • Кожен термінал реєструється в GK через RAS (UDP 1719)
  • GK призначає псевдонім → IP-адресу (alias translation)
  • Кілька зон = міжзонова маршрутизація через ієрархію GK
Зона є аналогом SIP-домену. Gatekeeper — аналогом SIP Registrar + Proxy.

Функції Gatekeeper

  • Address Translation — E.164 → IP
  • Admissions Control — ARQ/ACF/ARJ
  • Bandwidth Control — BRQ/BCF
  • Zone Management — реєстрація/відкликання
  • Маршрутизація сигналізації (Routed Mode)
  • Авторизація та тарифікація (CDR)
  • Балансування навантаження між MCU

📞 H.225 — Сигналізація викликів (Q.931)

Характеристики H.225

  • Базується на Q.931 (ISDN-сигналізація)
  • Транспорт: TCP, порт 1720
  • Встановлення та завершення викликів
  • Перенесення H.245 адреси (IP:порт для управління)
  • Підтримка Fast Connect (спрощений handshake)
  • Кодування: ASN.1 / PER (бінарне)
Fast Connect (Tunneling) — H.245 повідомлення передаються всередині H.225, що скорочує кількість round-trip до 1–2.

Основні повідомлення Q.931

🎛️ H.245 — Протокол управління медіа

H.245 — Control Protocol для узгодження медіа-можливостей та відкриття логічних каналів (RTP-потоків). Транспорт: TCP, динамічний порт, адреса передається в H.225 Setup/Connect.

Фази H.245

  1. Master/Slave Determination — визначення ролей (хто «Master»)
  2. Capabilities Exchange — обмін підтримуваними кодеками та можливостями
  3. Open Logical Channel (OLC) — відкриття RTP-каналу (IP:порт для медіа)
  4. Close Logical Channel (CLC) — закриття медіа-каналу

Ключові повідомлення H.245

MasterSlaveDeterminationВибір Master/Slave
OpenLogicalChannelВідкриття RTP-потоку
CloseLogicalChannelЗакриття RTP-потоку
FlowControlCommandКерування пропускною здатністю
EndSessionCommandЗавершення H.245-сесії

🔌 H.323 — Таблиця портів та вимоги до firewall

H.323 використовує множину TCP/UDP-портів — як фіксовані, так і динамічно узгоджені. Це ускладнює налаштування firewall та є причиною появи H.460 як NAT traversal рішення.
Порт / ДіапазонПротоколПризначенняПримітки
UDP 1719H.225 RASРеєстрація, допуск, статусGK Discovery (multicast 224.0.1.41, ITU-T H.323)
TCP 1720H.225 Call Sig.Встановлення/завершення викликуQ.931 signaling; обов'язковий
TCP 1300H.225 TLSШифрована сигналізація (H.460.22)Опціонально
TCP дин.H.245Узгодження медіа-можливостейПорт передається в H.225 Setup
UDP 49152–65535RTP / RTCPАудіо/відео медіа-потокиПорт узгоджується в H.245 OLC
TCP 1718GatekeeperGK Discovery (unicast альтернатива)Рідко використовується
Проблема для firewall: динамічні порти H.245 і RTP не можна заздалегідь дозволити. Потрібен ALG (Application Layer Gateway) або H.460.18/19, або SBC для NAT/firewall traversal.
Мінімальний набір портів для firewall: TCP 1720 вхідний + UDP 1719 двосторонній + дозволити встановлені TCP-з'єднання (stateful). Для повного функціонування — або ALG, або H.460.

📞 H.323 Call Flow — через Gatekeeper

  Terminal A          Gatekeeper           Terminal B
      │                   │                    │
      │── RAS: ARQ ──────►│                    │  ① запит допуску
      │◄─ RAS: ACF ───────│                    │  ② GK дозволяє
      │                   │                    │
      │── H.225 Setup ────────────────────────►│  ③ виклик (Direct Mode)
      │              (або через GK — Routed)   │
      │◄─ H.225 Alerting ──────────────────────│
      │                                        │
      │◄═══════════ H.245 (TCP) ══════════════►│  ④ узгодження медіа
      │◄═══════════ RTP Media ════════════════►│  🎤 аудіо/відео
      │                                        │
      │── RAS: DRQ ──────►│                    │  ⑤ сповіщення про кінець
      │◄─ RAS: DCF ───────│                    │
Перед встановленням виклику GK виконує Admission Control (ARQ/ACF) — перевірку прав та доступної смуги. Без ACF термінал не повинен ініціювати H.225 Setup. Після завершення виклику DRQ повідомляє GK для оновлення статистики та звільнення ресурсів.

Direct Mode (Direct Call Signaling)

H.225 Setup надсилається безпосередньо від A до B. GK лише перевіряє допуск через RAS. Менший контроль, менше навантаження на GK.

Routed Mode (GK-Routed Signaling)

Вся H.225-сигналізація проходить через GK. GK має повний контроль, може вести CDR, застосовувати політики. Аналог SIP Proxy.

🖥️ H.239 — Подвійний потік (Dual Stream)

H.239 — розширення H.323, що дозволяє одночасно передавати два відеопотоки: відео зустрічі (camera) та контент (презентація, робочий стіл, документи).

Принцип роботи

  • Визначає два окремих логічних RTP-канали
  • Live Channel — відео з камери (people video)
  • Presentation Channel — контент (slides, screen)
  • Узгоджується через H.245 ExtendedVideoCapability
  • Застосовується у системах Polycom, Cisco Tandberg, LifeSize

📡 RTP/RTCP — Транспорт медіа в H.323

RTP (Real-time Transport Protocol, RFC 3550) — протокол доставки аудіо/відео в реальному часі. Порти узгоджуються в H.245 OpenLogicalChannel і не є фіксованими, що є ключовою відмінністю від сигналізації.

Структура RTP

  • SSRC — унікальний ідентифікатор джерела медіа
  • PT (Payload Type) — тип кодека (G.711=0/8, G.729=18, H.264=динамічний)
  • Sequence Number — порядковий номер для реасемблювання
  • Timestamp — час семплювання (синхронізація відео+аудіо)
  • CSRC — список джерел при мікшуванні MCU
RTP = UDP + {seq, ts, ssrc, payload}. Без гарантії доставки — втрати пакетів = артефакти звуку.

RTCP — управління якістю

  • SR (Sender Report) — статистика відправника: bytes, packets, NTP timestamp
  • RR (Receiver Report) — якість отримання: втрати, jitter, RTT
  • SDES — опис джерела (CNAME, NAME)
  • BYE — завершення сесії
Безпека RTP: без H.235.6/SRTP RTP-потоки передаються у відкритому вигляді. Будь-хто у мережі може прослухати дзвінок або впровадити фальшивий RTP (injection).

🌐 H.460 — NAT Traversal у H.323

H.460.17

  • RAS через TCP (замість UDP)
  • Допомагає при NAT, що блокує UDP
  • Тунелювання RAS через H.225 TCP

H.460.18

  • Keepalive пакети для підтримки NAT-сесії
  • Нотифікація GK про IP/порт термінала
  • Сигналізація через вхідне з'єднання

H.460.19

  • NAT traversal для медіа (RTP)
  • Мультиплексування RTP/RTCP на одному порту
  • Сумісний з H.460.18

H.460.24

  • Підтримка IPv4/IPv6 dual-stack
  • NAT traversal для змішаних мереж
На практиці H.460.18/19 широко підтримується Cisco, Polycom та Tandberg. Для складних NAT часто застосовується TURN-подібний релей або SBC.

⚡ H.323 Fast Connect — оптимізація встановлення виклику

Fast Connect (H.323 Annex E) — механізм скорочення затримки встановлення виклику. Замість окремого H.245 TCP-з'єднання, параметри медіа-каналів передаються всередині H.225 Setup/Connect.

Стандартний режим (без Fast Connect)

A ──── H.225 Setup ──────────────► B
A ◄─── H.225 Alerting/Connect ──── B
  [відкриття H.245 TCP]
A ──── H.245 MSD ────────────────► B
A ◄─── H.245 MSD Ack ─────────────B
A ──── H.245 CapabilitySet ──────► B
A ◄─── H.245 CapabilitySet ───────B
A ──── H.245 OLC (audio) ────────► B
A ◄─── H.245 OLC Ack ─────────────B
  [лише тепер починається медіа]
Затримка: 4–6 round-trips ≈ 200–600 мс (LAN)

Fast Connect (H.245 tunneled in H.225)

A ──── H.225 Setup ──────────────► B
        (містить FastStart:
         OLC для audio/video)
A ◄─── H.225 Connect ─────────────B
        (містить FastStart ACK:
         підтверджені канали)
  [медіа починається ОДРАЗУ]
Затримка: 1 round-trip ≈ 50–100 мс (LAN)
Безпека: при Fast Connect H.245 не виконує повний обмін можливостями — можливий downgrade до слабших кодеків. Рекомендується валідувати FastStart параметри на SBC.

🔒 H.235 — Безпека в H.323

H.235 — стандарт безпеки для H.323: аутентифікація, цілісність та конфіденційність сигналізації і медіа. Входить до стека H.323 починаючи з версії 2 (1998).

Профілі H.235

ПрофільОпис
H.235.1Базова безпека на основі симетричних ключів
H.235.2Аутентифікація на базі сертифікатів (PKI)
H.235.3Гібридна безпека (симетрична + PKI)
H.235.5Шифрування медіа (framework)
H.235.6Шифрування голосу: AES-128
H.235.8MIKEY для обміну ключами SRTP

Механізми H.235

  • MD5 / SHA-1 / SHA-256 — хешування для автентифікації
  • HMAC-SHA1 — цілісність повідомлень H.225/H.245
  • DH обмін ключами — Perfect Forward Secrecy
  • RSA / ECDSA — цифровий підпис (сертифікати)
  • AES-128/256-CBC — шифрування медіа
  • TLS — захист H.225 (H.460.22)
H.235 реалізовано значно складніше, ніж TLS у SIP. Взаємодія між різними вендорами часто проблематична.

⚠️ Загрози та атаки на H.323

Специфічні загрози H.323

  • Gatekeeper Spoofing — підробка GK для перехоплення трафіку
  • RAS Flooding — DoS-атака через лавину ARQ-запитів
  • Malformed ASN.1 — переповнення буфера через некоректні пакети
  • Call Hijacking — перехоплення виклику через фальшивий Connect
  • H.245 Downgrade — примушення до слабких кодеків/алгоритмів
  • RTP Injection — підміна медіа-потоку
  • Zone Bypass — обхід контролю GK через прямі виклики

Загальні VoIP-загрози

  • Eavesdropping — прослуховування незашифрованого RTP
  • Toll Fraud — несанкціоновані виклики
  • Replay Attack — відтворення перехоплених повідомлень
  • Man-in-the-Middle — атака посередника
  • Registration Hijacking — перехоплення реєстрації в GK
CVE-2004-0054, CVE-2007-5575 — відомі вразливості в реалізаціях H.323 (Cisco, OpenH323). Бінарний ASN.1 складніше аналізувати, ніж текстовий SIP.

🎯 STRIDE-модель загроз для H.323-інфраструктури

Застосування методології STRIDE (Microsoft) до H.323 дозволяє систематично ідентифікувати загрози та визначити пріоритети захисних заходів.

🔑 H.235 — Аутентифікація терміналів

Password-Based (H.235.1)

Terminal A              Gatekeeper
    │                       │
    │── RRQ ───────────────►│
    │   (без аутентифікації) │
    │◄─ RRJ (authChallenge) ─│  nonce + alg
    │                       │
    │── RRQ ───────────────►│
    │   cryptoToken:        │
    │   HMAC-SHA1(pwd,nonce)│
    │◄─ RCF ────────────────│  ✅ зареєстровано
Аналогічно до SIP Digest Auth, але з бінарним кодуванням та нестандартизованим форматом між вендорами.

Certificate-Based (H.235.2)

  • Термінал має X.509-сертифікат
  • GK перевіряє сертифікат через PKI
  • TLS-з'єднання для H.225 (порт 1300)
  • Взаємна автентифікація (mTLS)
  • Більш безпечний, але складний у розгортанні

Практичні рекомендації

  • Використовувати H.235.6 + AES-128 для медіа
  • TLS для H.225 (H.460.22)
  • Сегментація мережі (VLAN)
  • SBC на периметрі

🔀 H.323 Gateway — міжмережева взаємодія

H.323 Gateway — пристрій, що здійснює перетворення протоколів між H.323 та іншими телефонними мережами. Є обов'язковим компонентом для підключення до PSTN.

H.323 ↔ PSTN/ISDN

  • Перетворення H.225/H.245 ↔ ISDN Q.931
  • Транскодування кодеків (G.711 ↔ G.729)
  • E.164 ↔ IP-адреса
  • Інтерфейси: E1/T1, BRI, PRI

H.323 ↔ SIP

  • B2BUA або proxy між двома стеками
  • Відображення: INVITE ↔ Setup
  • SDP ↔ H.245 capabilities
  • Приклади: Cisco IOS, OpenH323

H.323 ↔ H.320 (ISDN)

  • Взаємодія з відеосистемами ISDN
  • Конвертація H.261 ↔ H.261
  • BRI/PRI інтерфейси
  • Застаріло, але ще присутнє
Безпека Gateway: шлюз є критичною точкою — потребує суворої автентифікації, захисту від toll fraud та моніторингу аномальних патернів дзвінків.

👥 MCU — Multipoint Control Unit

MCU (H.231/H.243) — компонент H.323 для проведення багатоточкових конференцій. Складається з MC (Multipoint Controller) та MP (Multipoint Processor).

Компоненти MCU

MC (Multipoint Controller)

  • Управляє конференцією (H.245)
  • Узгоджує можливості всіх учасників
  • Визначає спільний «lowest common denominator»

MP (Multipoint Processor)

  • Обробляє медіа (мікшування аудіо)
  • Перемикання відео (Voice-Activated Switching)
  • Транскодування між форматами

Режими конференції

РежимОпис
CentralizedВесь медіа через MCU (типовий)
DecentralizedМедіа між учасниками напряму (multicast)
HybridКомбінація Centralized + Decentralized
Сучасні MCU: Cisco Meeting Server, Pexip, Acano. Підтримують H.323 + SIP + WebRTC одночасно.

🚀 H.323 у сучасних мережах

Де H.323 ще використовується

  • Відеозали Cisco / Polycom / Tandberg — спадкове обладнання
  • Корпоративні IP-PBX зі старою інфраструктурою
  • Телемедицина (стандарти ITU-T для відеоконсультацій)
  • Дистанційна освіта (H.323 + H.239)
  • Урядові та оборонні системи зв'язку
  • Мережі оператора класу (carrier-grade GW)

Сучасний стан

Міграція до SIP/WebRTC

  • Cisco замінює H.323 на SIP у нових продуктах
  • Microsoft Teams / Lync (SIP-based) підтримують H.323 через спеціальний коннектор
  • Polycom → SIP-firmware update

Гібридні рішення

  • H.323/SIP Interop Gateway (Cisco CME, CUBE)
  • WebRTC ↔ H.323 шлюзи (Pexip, Cisco CMS)
  • Microsoft Teams: H.323/SIP connector
З точки зору кібербезпеки, спадкові H.323-вузли — критична зона ризику: застаріле ПЗ рідко отримує патчі, а бінарне кодування ASN.1 ускладнює контроль. Гібридні середовища (H.323+SIP) збільшують поверхню атаки та потребують окремого SBC на стику.

🛡️ Захист H.323-інфраструктури

Мережеві заходи

  • Firewall — обмежити TCP 1720, UDP 1719, TCP 1300 (TLS)
  • SBC (Session Border Controller) — топологічне приховування, нормалізація ASN.1
  • VLAN — ізоляція голосового трафіку від даних
  • IDS/IPS — сигнатури для H.323-атак (Snort rules)
  • Rate limiting — ARQ/RRQ flood захист
  • ACL — білі списки IP для реєстрації в GK

Протокольні заходи

  • H.235.6 — шифрування медіа (AES-128)
  • TLS для H.225 (H.460.22, порт 1300)
  • H.235.2 — PKI-аутентифікація терміналів
  • GK-Routed mode — централізований контроль
  • Моніторинг CDR на аномалії (toll fraud)
  • Регулярне оновлення прошивки обладнання
H.323 протоколи складніші для аналізу IDS через бінарне кодування ASN.1. Рекомендується глибока перевірка пакетів (DPI) через SBC.

🎯 H.323 у контексті кібербезпеки

Відомі CVE та вразливості

  • CVE-2004-0054 — переповнення буфера в ASN.1 парсері Cisco IOS (CVSS 10.0)
  • CVE-2003-0819 — DoS у Microsoft NetMeeting (H.323)
  • CVE-2007-5570 — обхід аутентифікації GK
  • CVE-2012-0393 — вразливість у Cisco H.323 реалізації
  • Fuzzing ASN.1 — стабільне джерело нових вразливостей

Пентест H.323-інфраструктури

  • Reconnaissance: сканування TCP 1720, UDP 1719 (nmap -sU -p 1719)
  • GRQ flooding: пошук GK у мережі через broadcast GRQ
  • RAS fuzzing: інструменти Boofuzz, ASN.1-фаззери
  • Call interception: MITM при відсутності H.235
  • Wireshark decode: аналіз RTP при незашифрованих потоках
Бінарний формат ASN.1 ускладнює виявлення атак традиційними IDS. Спеціалізовані H.323-декодери є у Wireshark та Snort (preprocessor h323).
