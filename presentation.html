🔊 Що таке VoIP?

VoIP (Voice over Internet Protocol) — технологія передачі голосу у реальному часі через IP-мережі замість традиційних телефонних ліній (PSTN).

Традиційна телефонія (PSTN)

  • Комутація каналів (circuit switching)
  • Виділений фізичний канал на час розмови
  • 64 кбіт/с на виклик (G.711)
  • Висока надійність, фіксована затримка
  • Висока вартість міжнародних дзвінків

VoIP

  • Комутація пакетів (packet switching)
  • Спільне використання смуги пропускання
  • 8–100+ кбіт/с залежно від кодека
  • Змінна затримка (jitter), потрібен QoS
  • Значно нижча вартість, гнучкість

🔵 SIP — Session Initiation Protocol

SIP (RFC 3261, 2002) — текстовий протокол сигналізації рівня застосунків для встановлення, модифікації та завершення мультимедійних сесій.

Ключові характеристики

  • Базується на HTTP/SMTP-семантиці
  • Транспорт: UDP / TCP / TLS / SCTP
  • Порт за замовчуванням: 5060 (UDP/TCP), 5061 (TLS)
  • Текстовий формат — легко читати і налагоджувати
  • Розширюваний: SUBSCRIBE, NOTIFY, REFER…
  • Незалежний від медіа (лише сигналізація)

Функції SIP

  • Реєстрація UA (REGISTER)
  • Встановлення виклику (INVITE)
  • Локалізація кінцевого пристрою
  • Визначення медіа-можливостей (SDP)
  • Управління сесією (hold, transfer, conference)
  • Завершення сесії (BYE)
  • Миттєві повідомлення (MESSAGE)

🧩 Компоненти SIP-мережі

👤 User Agent (UA)

  • UAC (Client) — ініціює запити
  • UAS (Server) — приймає та відповідає
  • Приклади: Zoiper, Linphone, Cisco IP Phone

🔀 Proxy Server

  • Маршрутизує SIP-запити від UAC до UAS
  • Stateful / Stateless
  • Може розгалужувати виклик (forking)

📋 Registrar Server

  • Обробляє REGISTER-запити
  • Зберігає прив'язки AOR → контакт (IP)
  • AOR = sip:user@domain

🔁 Redirect Server

  • Відповідає кодом 3xx
  • Вказує UA новий URI для запиту
B2BUA (Back-to-Back User Agent) — одночасно є UAS і UAC; типовий для IP PBX (Asterisk, FreePBX, 3CX). Повністю контролює сесію.

🔗 SIP URI та адресація

sip:alice@atlanta.com               ← базовий URI
sip:alice@192.0.2.4:5060           ← IP + порт
sip:alice@atlanta.com;transport=tcp ← параметр транспорту
sips:alice@atlanta.com             ← захищений TLS URI
tel:+380441234567                   ← номерний URI (RFC 3966)

Заголовки адресації

  • From: — адреса відправника
  • To: — адреса одержувача
  • Contact: — поточна IP-адреса UA
  • Via: — шлях запиту (hop-by-hop)
  • Route / Record-Route: — примусова маршрутизація

Ідентифікація діалогу

  • Call-ID: — унікальний ID сесії
  • CSeq: — порядковий номер запиту
  • branch у Via — ідентифікатор транзакції
  • tag у From/To — ідентифікатор діалогу
  • Діалог = Call-ID + From-tag + To-tag

📞 SIP Call Flow — базовий дзвінок

  Alice (UAC)         SIP Proxy          Bob (UAS)
      │                   │                  │
      │─── INVITE ────────►│                  │
      │                   │─── INVITE ───────►│
      │◄── 100 Trying ────│◄── 100 Trying ───│
      │◄── 180 Ringing ───│◄── 180 Ringing ──│   🔔 дзвінок у Bob
      │                   │◄── 200 OK ───────│   ✅ Bob підняв трубку
      │◄── 200 OK ────────│                  │
      │─── ACK ───────────────────────────►│   підтвердження
      │                                      │
      │◄══════════════ RTP Media ════════════►│   🎤 двостороннє аудіо
      │                                      │
      │─── BYE ───────────────────────────►│   завершення розмови
      │◄── 200 OK ────────────────────────│
Сигналізація (SIP) і медіа (RTP) передаються незалежно. INVITE містить SDP-пропозицію (offer), 200 OK — SDP-відповідь (answer).

📝 Реєстрація SIP UA (REGISTER)

  Alice UA                     Registrar Server
      │                               │
      │─── REGISTER ─────────────────►│
      │    From: sip:alice@domain.com  │
      │    To:   sip:alice@domain.com  │
      │    Contact: sip:alice@10.0.0.5 │
      │    Expires: 3600               │
      │                               │
      │◄── 401 Unauthorized ──────────│  WWW-Authenticate: Digest realm=...
      │                               │
      │─── REGISTER ─────────────────►│  Authorization: Digest response=...
      │                               │
      │◄── 200 OK ────────────────────│  ✅ зареєстровано
Digest Authentication (RFC 3261 §22): сервер надсилає nonce, клієнт відповідає хешем MD5(HA1:nonce:HA2). У сучасних системах рекомендується SHA-256 (RFC 7616). Для надійного захисту реєстрацію слід виконувати по TLS.

🎵 RTP та RTCP (RFC 3550)

RTP — Real-time Transport Protocol

  • Передача аудіо/відео пакетів у реальному часі
  • Транспорт: UDP (парні порти)
  • Заголовок: payload type, sequence number, timestamp, SSRC
  • Не гарантує доставку або порядок
  • SRTP (RFC 3711) — шифрована версія

RTCP — RTP Control Protocol

  • Статистика сесії: втрати, jitter, RTT
  • Порт = RTP-порт + 1
  • SR (Sender Report) / RR (Receiver Report)
  • SDES — опис джерела (CNAME)
  • Основа для адаптивного бітрейту

📋 SDP — Session Description Protocol (RFC 4566)

Структура SDP

v=0                          ← версія
o=alice 123 456 IN IP4 10.0.0.1 ← автор/сесія
s=VoIP Call                  ← назва сесії
c=IN IP4 10.0.0.1            ← з'єднання
t=0 0                        ← час

m=audio 49170 RTP/AVP 0 8 97 ← медіа-рядок
a=rtpmap:0 PCMU/8000         ← G.711 µ-law
a=rtpmap:8 PCMA/8000         ← G.711 A-law
a=rtpmap:97 iLBC/8000        ← iLBC кодек
a=sendrecv                   ← двосторонній режим

m=video 51372 RTP/AVP 96
a=rtpmap:96 H264/90000
a=fmtp:96 profile-level-id=42e01f

Offer / Answer модель (RFC 3264)

  • Offer: INVITE містить список підтримуваних кодеків
  • Answer: 200 OK — вибирає один або кілька
  • Визначає IP-адресу та порт для RTP
  • ICE-кандидати для NAT traversal
  • Режими: sendrecv, sendonly, recvonly, inactive
SDP описує що передавати і куди.
RTP виконує фактичну передачу медіа.

🌐 NAT Traversal: STUN / TURN / ICE

STUN (RFC 8489)

  • UA дізнається свою публічну IP:port
  • Легкий протокол, низька затримка
  • Не вирішує symmetric NAT
  • Типово: Google STUN, Coturn

TURN (RFC 8656)

  • Relay-сервер ретранслює медіа
  • Вирішує всі типи NAT
  • Вища затримка, навантаження на сервер
  • Резервний варіант за ICE

ICE (RFC 8445)

  • Комбінує STUN та TURN
  • Збирає host / srflx / relay кандидати
  • Connectivity checks (STUN binding)
  • Вибирає найкращий шлях
  UA (private)          STUN/TURN Server         UA (private)
  10.0.0.5:49170   ──► 203.0.113.1:3478 ◄──   192.168.1.10:51234
       │                      │                       │
       └─── ICE: перевірка кандидатів ────────────────┘
            1. host:  10.0.0.5:49170
            2. srflx: 198.51.100.2:12345  (через STUN)
            3. relay: 203.0.113.1:40000   (через TURN)

⚡ Якість обслуговування (QoS)

Ключові параметри

  • Затримка (latency): < 150 мс one-way (рек. ITU G.114)
  • Jitter: < 30 мс (згладжується jitter buffer)
  • Втрати пакетів: < 1% для задовільної якості
  • Пропускна здатність: 8–100 кбіт/с на виклик
Затримка > 300 мс суттєво погіршує розбірливість мови та заважає природній розмові.

Механізми QoS

  • DSCP/DiffServ: маркування пакетів (EF = 46 для голосу)
  • 802.1p CoS: пріоритизація на рівні L2
  • VLAN: ізольована голосова мережа
  • Traffic policing/shaping
  • Jitter buffer: adaptive / fixed
  • PLC (Packet Loss Concealment): приховування втрат
  • AEC: придушення акустичного відлуння

🔒 Безпека VoIP

Основні загрози

  • Toll fraud — несанкціоновані дзвінки
  • Eavesdropping — прослуховування RTP
  • SIP flooding / DoS
  • Registration hijacking
  • SPIT — спам через IP-телефонію
  • Man-in-the-Middle
  • Replay attacks
  • Vishing — голосовий фішинг

Захисні заходи

  • TLS для сигналізації SIP (порт 5061)
  • SRTP/ZRTP для шифрування медіа
  • SIP Digest Auth + SHA-256 (RFC 7616)
  • SBC (Session Border Controller)
  • Fail2ban + IP filtering
  • VPN для корпоративних з'єднань
  • STIR/SHAKEN — верифікація CLI (RFC 8226)
  • Rate limiting REGISTER/INVITE
SBC (Session Border Controller) — ключовий елемент захисту: topology hiding, rate limiting, TLS/SRTP termination, DoS-захист, нормалізація SIP.

🛡️ SRTP — Secure RTP (RFC 3711)

Особливості SRTP

  • Шифрування: AES-CTR (128 або 256 біт)
  • Аутентифікація: HMAC-SHA1
  • Захист від відтворення (replay protection)
  • Мінімальні накладні витрати (~10 байт)
  • Управління ключами: SDES, DTLS-SRTP
DTLS-SRTP (RFC 5764) — рекомендований метод обміну ключами у WebRTC. Забезпечує Perfect Forward Secrecy (PFS).

🌍 WebRTC та сучасний VoIP-стек

WebRTC — відкритий стандарт W3C/IETF для браузерних медіа-комунікацій без плагінів (Chrome, Firefox, Safari, Edge).

Протокольний стек WebRTC

  • Сигналізація: не стандартизована (SIP/WebSocket/proprietary)
  • JSEP: JavaScript Session Establishment Protocol
  • ICE/STUN/TURN: NAT traversal
  • DTLS: обмін ключами (PFS)
  • SRTP: шифрування медіа
  • SCTP over DTLS: DataChannel
  • Opus / VP8/VP9/H.264/AV1: кодеки

Застосування

  • Google Meet, Microsoft Teams
  • Discord, Zoom (WebRTC-режим)
  • WhatsApp Web, Telegram Web
  • Браузерні контакт-центри
  • IoT голосові пристрої
SIP over WebSocket (RFC 7118) — SIP UA у браузері без встановлення застосунку.

  • Wireshark — аналіз SIP/RTP пакетів
  • sngrep — консольний SIP-монітор
  • SIPp — генератор навантаження SIP
  • Homer SIPCapture — корпоративний моніторинг
  • VoIPmonitor — аналіз MOS у реальному часі
  • tcpdump + pcap — захоплення трафіку

📌 Підсумки

  • VoIP замінює PSTN завдяки економічності та гнучкості
  • SIP (RFC 3261) — домінуючий протокол сигналізації
  • Чіткий поділ: сигналізація (SIP) ↔ медіа (RTP)
  • SDP описує медіа-параметри (offer/answer)
  • ICE/STUN/TURN вирішують проблему NAT
  • TLS + SRTP — обов'язковий мінімум безпеки
  • QoS критичний: затримка < 150 мс, jitter < 30 мс
  • WebRTC виносить VoIP у будь-який браузер
