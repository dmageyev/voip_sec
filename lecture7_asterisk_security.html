🔺 CIA Тріада в контексті IP-АТС

CIA Triad — фундаментальна модель інформаційної безпеки: Confidentiality · Integrity · Availability.
ВластивістьЗагроза для IP-АТСЗахист в Asterisk
Confidentiality (Конфіденційність)Перехоплення SIP/RTP, eavesdroppingTLS для сигналізації, SRTP для медіа
Integrity (Цілісність)Підміна SIP-запитів, Caller ID spoofingDigest Auth, TLS з верифікацією сертифіката
Availability (Доступність)DoS/DDoS на SIP-порт, flood реєстраційfail2ban, rate limiting, VLAN-ізоляція

Практичне застосування:

  • Порушення C → витік записів переговорів, комерційної таємниці
  • Порушення I → шахрайські дзвінки від імені легітимного номера
  • Порушення A → АТС недоступна, бізнес-процеси зупинено
Захист IP-АТС = забезпечення всіх трьох властивостей одночасно.

Authentication

«Що можна?»

🔒 Fail-Safe Defaults: Безпечна відмова

Принцип: система за замовчуванням забороняє все, а дозволяє лише явно налаштоване. Якщо перевірка неможлива або виникла помилка — відмовити у доступі.
❌ Fail-Open (небезпечний):
дозволити все → заблокувати відоме шкідливе
; Небезпечний дефолт
allowguest=yes   ; анонімні дзвінки!
permit=0.0.0.0/0 ; дозволити всіх
deny=10.0.0.0/8  ; заблокувати частково
context=default  ; відкритий контекст
✅ Fail-Secure (правильно):
заборонити все → дозволити явно перевірене
; Безпечний дефолт
allowguest=no    ; тільки auth!
deny=0.0.0.0/0   ; заборонити всіх
permit=192.168.10.0/24 ; дозволити свою
context=restricted ; мінімальний
Наслідок порушення: allowguest=yes + відсутність ACL = Toll Fraud без злому паролів.

🎯 Мінімізація поверхні атаки

Attack Surface — сукупність усіх точок входу, через які зловмисник може взаємодіяти з системою. Принцип: зменшити кількість та розмір цих точок до мінімально необхідного.

Повна поверхня (за замовчуванням):

UDP/5060  — SIP (відкритий для всіх)
UDP/5061  — TLS-SIP
UDP/10000-20000 — RTP (широкий)
TCP/5038  — AMI (відкритий)
TCP/8088  — HTTP/ARI
+ всі завантажені модулі

Мінімізована поверхня:

UDP/5060 — тільки внутр. підмережа
UDP/5061 — тільки trusted trunk
UDP/RTP  — обмежений діапазон
TCP/5038 — тільки loopback
TCP/8088 — вимкнений або HTTPS

Вимкнення модулів (modules.conf):

[modules]
autoload=no            ; НЕ завантажувати все підряд
load => chan_pjsip.so  ; тільки необхідне
load => pbx_config.so
load => app_dial.so
; chan_sip.so — НЕ завантажуємо без потреби
; res_http_websocket.so — тільки для WebRTC

🔐 Zero Trust для IP-АТС

Zero Trust: «Ніколи не довіряй, завжди перевіряй» — жоден компонент не є автоматично довіреним, навіть всередині мережі.

Традиційна модель (периметр):

Internet  │  LAN
  ❌      │  ✅ (всі довірені)
          │
       Asterisk
  ← SIP без auth від
    внутрішніх IP
Компромет одного пристрою у LAN → доступ до АТС без перешкод

Zero Trust:

Internet  │  LAN
  ❌      │  перевіряється!
          │
       Asterisk
  ← SIP Digest Auth
    від КОЖНОГО IP
Мікросегментація: навіть VLAN-сусіди не довіряємо без auth
[6001]
type=endpoint
auth=auth6001     ; пароль для КОЖНОГО endpoint
acl=internal_acl  ; + перевірка IP
context=internal  ; + обмежений контекст

🔐 Принцип мінімізації привілеїв (PoLP) → Asterisk

Principle of Least Privilege (Saltzer & Schroeder, принцип №6): кожен суб'єкт отримує рівно ті права, які необхідні для виконання своєї функції, — і не більше.
┌──────────────────────────────────────────────────────────┐
│  Рівень 4 — Диалплан                                     │
│  Принцип: кожен контекст дозволяє тільки потрібні        │
│  напрямки (whitelist замість blacklist)                   │
├──────────────────────────────────────────────────────────┤
│  Рівень 3 — Asterisk ACL (permit/deny по IP)             │
│  Принцип: кожен endpoint бачить тільки свій ACL          │
├──────────────────────────────────────────────────────────┤
│  Рівень 2 — Мережа (VLAN, iptables, fail2ban)            │
│  Принцип: сегментація — кожна зона має мінімальний доступ│
├──────────────────────────────────────────────────────────┤
│  Рівень 1 — ОС (непривілейований користувач asterisk)    │
│  Принцип: процес не може робити більше, ніж його user    │
└──────────────────────────────────────────────────────────┘
Defense-in-Depth: кожен рівень PoLP = окремий рубіж. Компрометація одного рівня не розкриває всю систему.

🐧 Захист на рівні операційної системи

Запуск від непривілейованого користувача:

# Створення системного користувача
useradd -r -s /sbin/nologin asterisk
groupadd asterisk

# Запуск з обмеженими правами
asterisk -U asterisk -G asterisk

systemd unit-файл — захисні директиви:

[Service]
User=asterisk
Group=asterisk
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true
CapabilityBoundingSet=CAP_NET_BIND_SERVICE
Навіщо? Якщо зловмисник виконає код через AGI — він не отримає root-доступу.

Що захищають директиви systemd:

  • NoNewPrivileges — заборона підвищення прав
  • ProtectSystem=strict — ФС тільки для читання
  • ProtectHome=true — немає доступу до /home
  • PrivateTmp=true — ізольована /tmp
  • CapabilityBoundingSet — тільки потрібні capabilities

Архітектура VLAN для IP-АТС:

  • Internet → PBX: забороненo без явного дозволу

🛡️ ACL: Рівні захисту (Complete Mediation)

Defense-in-Depth для IP-фільтрації:

Incoming SIP INVITE
        │
        ▼
┌───────────────────┐
│  1. iptables/nft  │ → блокує цілі підмережі/країни
└────────┬──────────┘
         ▼
┌───────────────────┐
│  2. fail2ban      │ → динамічно блокує brute-force IP
└────────┬──────────┘
         ▼
┌───────────────────┐
│  3. Asterisk ACL  │ → permit/deny на рівні конфігурації
└────────┬──────────┘
         ▼
┌───────────────────┐
│  4. Контекст      │ → обмежує доступні extension
└───────────────────┘
Реалізація принципу Complete Mediation: кожен запит перевіряється на кожному рівні незалежно.

🔥 fail2ban + iptables/nftables

Конфігурація fail2ban для Asterisk:

; /etc/fail2ban/filter.d/asterisk.conf
[Definition]
failregex = NOTICE.* <HOST>.*: Registration from .* failed
            NOTICE.* <HOST>.*: No matching peer found
            WARNING.* <HOST>.*: Digest auth failed
            NOTICE.* <HOST>.*: Sending fake auth rejection

; /etc/fail2ban/jail.local
[asterisk]
enabled  = true
port     = 5060,5061
filter   = asterisk
logpath  = /var/log/asterisk/messages
maxretry = 5
bantime  = 3600   ; блокування на 1 годину
findtime = 600    ; 5 спроб за 10 хвилин → бан

nftables rate-limiting для SIP:

nft add rule inet filter input \
  udp dport 5060 \
  limit rate 30/second \
  burst 50 packets accept

nft add rule inet filter input \
  udp dport 5060 drop
Логіка: перші 50 пакетів + 30/с → accept. Все понад — drop. Захист від SIP flood.
fail2ban-client status asterisk — перевірка поточних банів

🗂️ Контексти Asterisk: концепція

Контекст — іменований простір у extensions.conf, що містить набір правил обробки дзвінків (extension patterns). Реалізація принципу Separation of Privilege.

Зв'язок endpoint → context → extension:

SIP INVITE від 6001
        │
        ▼ (chan_pjsip визначає endpoint)
 context = "internal"
        │
        ▼ (Asterisk шукає в [internal])
 exten => 6002,1,Dial(PJSIP/6002)   ✓ знайдено
 exten => 9.,1,Dial(PJSIP/trunk)    ✓ дозволено

SIP INVITE від зовнішнього trunk
        │
        ▼
 context = "from-trunk"
        │
        ▼ (в [from-trunk] немає внутрішніх extension!)
 exten => s,1,Answer()              ✓ тільки IVR
 exten => 6001,1,Dial(PJSIP/6001)  ✓ маршрут до внутрішнього

🌲 Ієрархія контекстів: include

; extensions.conf

[restricted]           ; гості, лобі
include => internal_phones

[internal]             ; співробітники
include => internal_phones
include => outbound_local

[internal_premium]     ; керівники
include => internal
include => outbound_international

[from-trunk]           ; вхідні від провайдера
exten => s,1,Goto(ivr,s,1)
exten => +380XXXXXXXXX,1,Dial(PJSIP/6001)

; ❌ КРИТИЧНА ПОМИЛКА:
; include => internal  ← в [from-trunk] → Toll Fraud!
Правило: include передає права вниз. Ніколи не включайте привілейований контекст у менш захищений.

Ієрархія прав:

restricted
  ↓ include
internal
  ↓ include
internal_premium

Best practice: явний дозвіл, заборона за замовчуванням (Fail-Safe Default).

💥 Загрози на рівні диалплану

Toll Fraud

Зловмисник реєструється (або обходить автентифікацію) і дзвонить на дорогі номери за рахунок жертви.

DISA (Direct Inward System Access)

Вхід у АТС ззовні з відкритим виходом у зовнішні лінії через введення PIN.

Voicemail → Outbound

VM-меню дозволяє передзвонити на довільний номер з зовнішньої лінії АТС.

AGI/System() Injection

Некоректна валідація вводу в AGI-скриптах або System() — виконання системних команд.

IVR DTMF Abuse

Ін'єкція через DTMF-введення в IVR для переходу на заборонені extension.

Всі загрози = наслідок порушення AAA, Fail-Safe Defaults або відсутності валідації вводу.

💸 Toll Fraud: Механізм атаки

1. Розвідка:
   svmap -r 5060 target_ip    # сканування SIP
   svwar -e100-200 target_ip  # перебір extension

2. Brute-force паролів:
   svcrack -u 6001 target_ip  # підбір секрету

3. Реєстрація скомпрометованого акаунта:
   REGISTER sip:6001@target ...
   Authorization: Digest ...

4. Генерація дорогих дзвінків:
   INVITE sip:+1900XXXXXXX@target ...
   INVITE sip:+441XXXXXXXXX@target ...

5. Збиток: сотні дзвінків за кілька годин
   → рахунок від провайдера = тисячі доларів
Типові цілі:
+1900 (преміум США)
+44 (Великобританія)
+371 (Латвія)
Міжнародні premium-rate

Уражені принципи:

  • Authentication (AAA) — слабкий пароль
  • Authorization — відсутня ізоляція контекстів
  • Fail-Safe Default — allowguest=yes
  • Availability (CIA) — фінансовий збиток → відключення послуги

✅ Валідація введення в диалплані

❌ НЕБЕЗПЕЧНО: довільне введення
exten => s,1,Read(USERNUM)
exten => s,2,Dial(PJSIP/trunk/${USERNUM})
; Будь-який номер → trunk!
❌ НЕБЕЗПЕЧНО: System() з змінною
exten => s,1,System(echo \
  ${CALLERID(num)} >> /tmp/log)
; Command injection!
✅ БЕЗПЕЧНО: FILTER() + перевірка довжини
exten => s,1,Read(USERNUM)
exten => s,2,Set(CLEAN=\
  ${FILTER(0-9,${USERNUM})})
exten => s,3,GotoIf(\
  $[${LEN(${CLEAN})} < 4]?invalid)
exten => s,4,GotoIf(\
  $[${LEN(${CLEAN})} > 15]?invalid)
exten => s,5,Goto(outbound,${CLEAN},1)
exten => s,n(invalid),Playback(invalid)
exten => s,n,Hangup()
✅ БЕЗПЕЧНО: фіксований аргумент
exten => s,1,System(/usr/local/bin/log.sh)

🚫 Захист від DISA

DISA (Direct Inward System Access): Зовнішній дзвінок → АТС → DISA() → введення PIN → отримання dial tone → дзвінок на будь-який номер.

Якщо DISA необхідна — мінімальний захист:

exten => s,1,DISA(no-password,restricted)
;             ↑ файл з PIN   ↑ обмежений ctx

; /etc/asterisk/disa.conf:
; 7491823654|restricted  ; складний PIN
DISA без PIN або зі слабким PIN = відкритий trunk для будь-кого.

Кращий підхід — замінити DISA на VPN:

; Замість:
Зовн. дзвінок → DISA → trunk

; Краще:
VPN-підключення
  → internal context
    → SIP-дзвінок
VPN забезпечує Confidentiality + Integrity. DISA — лише слабку Authentication.

📞 Обмеження одночасних дзвінків

Захист від масового Toll Fraud після компрометації акаунта:

; Обмеження одночасних каналів на акаунт
exten => _X.,1,Set(GROUP()=\
  outbound_${CALLERID(num)})
exten => _X.,2,Set(COUNT=${GROUP_COUNT(\
  outbound_${CALLERID(num)})})
exten => _X.,3,GotoIf(\
  $[${COUNT} > 3]?toolimit)
exten => _X.,4,Dial(PJSIP/trunk/${EXTEN})
exten => _X.,n(toolimit),Playback(conf-locked)
exten => _X.,n,Hangup(17)  ; User Busy

; Глобальне обмеження на trunk
[from-trunk-outbound]
exten => _X.,1,Set(GROUP()=trunk_out)
exten => _X.,2,GotoIf(\
  $[${GROUP_COUNT(trunk_out)} > 30]?overlimit)

У pjsip.conf endpoint:

[6001]
type=endpoint
max_contacts=2  ; ≤ 2 реєстрацій
Логіка: навіть якщо зловмисник отримав доступ до акаунта, він не зможе згенерувати більше 3 одночасних вихідних дзвінків. Обмеження збитку.

🔐 TLS та SRTP в Asterisk

Налаштування TLS (pjsip.conf):

[transport-tls]
type=transport
protocol=tls
bind=0.0.0.0:5061
cert_file=/etc/asterisk/certs/server.crt
priv_key_file=/etc/asterisk/certs/server.key
ca_list_file=/etc/asterisk/certs/ca.crt
method=tlsv1_2       ; мінімум TLS 1.2

[6001]
type=endpoint
transport=transport-tls
media_encryption=srtp
media_encryption_optimistic=no
; ↑ не допускати незашифроване!

Генерація сертифікатів:

cd /usr/share/asterisk/scripts
./ast_tls_cert \
  -C pbx.company.com \
  -O "Company" \
  -d /etc/asterisk/certs
TLS захищає сигналізацію SIP (CIA: Confidentiality + Integrity).
SRTP захищає медіа-потік RTP.
media_encryption_optimistic=yes дозволяє незашифровані з'єднання як fallback — Fail-Open! Використовуйте тільки у крайньому разі.

[readonly_user]
